## 시작하기 및 계정 설정

<Accordion title="Q1: Dodo Payments 대시보드에 로그인하려면 어떻게 하나요?">
  A: 대시보드에 방문하여 환영 이메일에 공유된 로그인 이메일과 비밀번호를 입력하세요. 문제가 발생할 경우 지원 팀에 문의해 주세요.
</Accordion>

<Accordion title="Q2: 인증 프로세스를 완료하기 위해 어떤 문서가 필요하나요?">
  A: 인증에 필요한 문서는 유효한 정부 발급 신분증(신원 확인용)과 사업 등록 문서(사업 확인용)입니다. 자세한 내용은 [인증 프로세스 페이지](/miscellaneous/verification-process)를 확인하세요.
</Accordion>

<Accordion title="Q3: 내 인증이 거부되면 어떻게 되나요?">
  A: 인증이 거부되면 거부 사유가 포함된 이메일을 받게 됩니다. 문제를 해결하고 문서를 수정하여 재신청하려면 지원 팀에 문의할 수 있습니다. 필요할 경우 재제출 옵션은 인증 팀에서 활성화됩니다.
</Accordion>

<Accordion title="Q4: 인증 프로세스는 얼마나 걸리나요?">
  A: 인증 프로세스는 일반적으로 **24-72 근무 시간**이 소요됩니다. 그러나 추가 문서가 필요한 경우 프로세스가 더 오래 걸릴 수 있습니다.
</Accordion>

<Accordion title="Q5: 현재 내 사업을 위한 웹사이트가 없습니다. 대신 회사의 LinkedIn 페이지를 제출할 수 있나요?">
  A: 우리는 귀하의 웹사이트가 필요합니다. 귀하가 판매하는 제품의 종류를 이해하고 실시간 액세스를 위해 귀하의 사업을 확인하기 위해서입니다. 웹사이트가 없거나 웹사이트를 구축 중이라면 support@dodopayments.com으로 문의해 주세요.
</Accordion>

<Accordion title="Q6: 나는 개인이고 등록된 사업체가 없습니다. Dodo Payments를 여전히 사용할 수 있나요?">
  A: 네, 우리는 등록되지 않은 사업체도 지원합니다. Dodo Payments를 사용하기 위해 등록된 사업체가 필요하지 않습니다. 개인으로 온보딩하여 국제 결제를 문제없이 받을 수 있습니다. 등록되지 않은 사업체로서 신원 확인 양식과 사업 확인 양식을 제출해야 합니다.
</Accordion>

<Accordion title="Q7: 다른 사업체를 Dodo Payments와 통합하고 싶습니다. 새 계정을 만들어야 하나요?">
  A: 아니요, 하나의 계정으로 여러 사업체를 관리할 수 있습니다. 새 사업체를 추가하려면 대시보드의 왼쪽 하단 모서리에 있는 + 아이콘을 클릭하고 '사업 추가' 옵션을 선택하세요.
/miscellaneous/accounts#managing-multiple-businesses에서 더 많은 세부정보를 참조할 수 있습니다.
</Accordion>

<Accordion title="Q8: Dodo Payments를 내 시스템에 통합하기 전에 인증이 완료될 때까지 기다려야 하나요?">
  A: 아니요, 기다릴 필요가 없습니다. 테스트 모드를 사용하여 통합을 시작할 수 있으며, 인증이 성공적으로 완료되면 프로덕션에서 실시간 결제를 시작할 수 있습니다. 테스트 및 라이브 모드에 대한 자세한 내용은 여기에서 확인하세요,
/miscellaneous/test-mode-vs-live-mode
</Accordion>

<Accordion title="Q9: 라이브 모드 액세스를 얻으려면 어떻게 해야 하나요?">
  A: 설정 탭에서 신원 및 사업 확인 문서를 제출했는지 확인하세요. 이미 제출했으며 72시간이 지났다면, 같은 페이지에서 보류 사유를 확인하고 지원 팀에 문의하세요.
</Accordion>

<Accordion title="Q10: Dodo Payments에 제휴 프로그램이 있나요?">
  A: 네, 제휴 프로그램을 제공합니다!

  <Card title="제휴 프로그램에 대해 더 알아보기" icon="users" href="/features/affiliates">
    제휴 프로그램에 가입하는 방법, 커미션 비율 및 수익을 시작하는 방법을 알아보세요.
  </Card>

</Accordion>

<Accordion title="Q11: 내가 만든 제품/고객을 삭제하려면 어떻게 하나요?">
  A: 현재로서는 기록 유지를 위해 제품이나 고객을 삭제할 수 없습니다. 우리는 현재 활성 제품 섹션에서 제품을 제거할 수 있는 아카이브 제품 기능을 작업 중입니다.
</Accordion>

<Accordion title="Q12: 결제 페이지에 사용자 정의 UI를 어떻게 설정할 수 있나요?">
  A: 결제 페이지에 귀하의 사업 로고와 사업 이름을 추가할 수 있습니다. 이를 업로드하려면 대시보드의 사업 프로필 페이지를 방문하세요.
</Accordion>

<Accordion title="Q13: 체크아웃 페이지의 브랜딩을 업데이트할 수 있나요?">
  A: 네. 다음과 같이 할 수 있습니다:
  - 대시보드 → 설정 → 사업 설정으로 이동
  - 로고, 브랜드 이름 및 지원 이메일을 업데이트하세요.
  - 변경 사항은 정적 링크와 동적 링크 모두에 반영됩니다.
</Accordion>

<Accordion title="Q14: Dodo Checkout을 iframe에 삽입할 수 있나요?">
  A: 아니요. 호스팅된 체크아웃 흐름은 iframe에 삽입을 지원하지 않습니다.

  대신:
  - 사용자를 Dodo 체크아웃 URL로 리디렉션하세요.
  - 또는 우리의 오버레이 체크아웃 기능을 사용하세요. [오버레이 체크아웃 문서](/developer-resources/overlay-checkout)
</Accordion>

<Accordion title="Q15: Dodo Payments에 제품을 추가하지 않고 일회성 호스팅 체크아웃 결제를 생성할 수 있나요?">
  A: 아니요. Dodo Payments 시스템에서는 제품이 필수입니다.
</Accordion>

<Accordion title="Q16: 전자상거래 웹사이트를 지원하나요?">
  A: 아니요, 전자상거래 웹사이트는 지원하지 않습니다.
</Accordion>

<Accordion title="Q17: Dodo Payments에서 브랜드 간 전환이 가능한가요?">
  A: 네, 제품에 대해 브랜드 간 전환이 가능합니다. 대시보드에서 제품을 생성하거나 편집할 때 원하는 브랜드를 선택할 수 있습니다.
</Accordion>

<Accordion title="Q18: Dodo Payments에서 브랜드를 삭제하거나 브랜드 URL을 업데이트할 수 있나요?">
  A: 아니요, Dodo Payments는 현재 다음을 지원하지 않습니다:
  - **브랜드 삭제**: 브랜드가 사용되지 않는 경우 무시하면 됩니다. 계정이나 운영에 영향을 미치지 않습니다.
  - **브랜드 URL 변경 또는 업데이트**: 생성된 후에는 브랜드 URL을 수정할 수 없습니다. URL이 잘못되더라도 설정이나 기능에 영향을 미치지 않습니다.
</Accordion>

<Accordion title="Q19: Dodo Payments를 Shopify와 통합하여 물리적 제품을 판매할 수 있나요?">
  A: 아니요, Dodo Payments는 현재 물리적 상품을 지원하지 않으며 Shopify 플러그인을 제공하지 않습니다.
</Accordion>

## 인증, 세금 및 규정 준수

<Accordion title="Q20: 내 인증이 왜 이렇게 오래 걸리나요?">
  A: Dodo Payments는 일반적으로 인증을 처리하는 데 24-72 근무 시간이 소요됩니다. 그러나 지연이 발생할 수 있는 이유는 다음과 같습니다:
  - 높은 신청량
  - 주말 및 공휴일
  - 불완전하거나 잘못된 제출
  - 위험 또는 규정 준수를 위한 수동 검토

  <Note>
  72시간이 지났다면 이메일이나 Dodo Payments 대시보드에서 필요한 조치를 확인하세요. 또한 Discord에서 팀에 등록된 이메일을 DM하여 후속 조치를 요청할 수 있습니다.
  </Note>
</Accordion>

<Accordion title="Q21: W-8BEN 양식을 제출했지만 서명란이 제대로 작동하지 않습니다. 어떻게 해야 하나요?">
  A: 다음과 같은 적절한 PDF 편집기를 사용하세요:
  - Adobe Acrobat
  - Mac Preview (대부분 작동)
  - Smallpdf 또는 PDFescape (온라인 편집기)

  서명에 교차 표시가 있는 경우:
  - 서명 후 파일을 저장하고 있는지 확인하세요.
  - 도구가 서명 레이어를 평면화하고 있는지 확인하세요(일부 편집기는 그렇지 않습니다).

  <Tip>
  브라우저 미리보기에서 서명하는 것을 피하세요. 디지털 서명이 제대로 삽입되지 않을 수 있습니다.
  </Tip>
</Accordion>

<Accordion title="Q22: 잘못된 등록 유형(사업 vs 개인)을 선택했습니다. 어떻게 수정하나요?">
  A: 잘못된 법인 유형(예: “조직”을 선택했지만 개인이거나 GST가 있는 개인 사업자)을 선택한 경우 인증이 보류 중일 수 있으며, 문서를 변경하거나 업데이트해야 할 수 있습니다.

  수정 단계:
  1단계: Intercom 또는 Discord를 통해 지원 팀에 알리면 Payout Compliance 양식의 재제출을 활성화하는 데 도움을 줄 것입니다.
  2단계: 올바른 등록 유형을 선택하고 필요한 문서를 재제출하세요.
</Accordion>

<Accordion title="Q23: 내 인증이 법인 불일치로 거부되었습니다. 어떻게 해야 하나요?">
  A: 법인 불일치로 인증이 거부된 경우(예: "조직"을 선택했지만 개인이거나 GST가 있는 개인 사업자), 등록 유형을 수정하고 문서를 재제출해야 합니다.

  수정 단계:
  1단계: 대시보드나 이메일에서 거부 사유를 확인하세요.
  2단계: Intercom 또는 지원 이메일(support@dodopayments.com)을 통해 Payout Compliance 양식의 재제출 접근을 요청하세요.
  3단계: 올바른 등록 유형(개인 vs 조직)을 선택하고 모든 필요한 문서를 재제출하세요.
  4단계: 준수 팀이 재제출을 검토할 때까지 기다리세요.

  <Note>
  재제출 옵션은 지원 팀에 연락한 후 인증 팀에서 활성화됩니다. 이번에는 올바른 법인 유형을 선택하여 추가 지연을 피하세요.
  </Note>
</Accordion>

<Accordion title="Q24: $100의 수익을 달성했습니다. 내 계정이 더 빨리 인증되나요?">
  A: 네. 수익이 지급 기준($100)을 초과하면 Dodo Payments는 귀하의 인증을 우선 처리합니다.

  <Note>
  이는 즉각적인 승인을 보장하지 않습니다. 귀하의 사업 모델과 문서가 규정을 준수하는지 확인하세요.
  </Note>

  $100를 초과했지만 여전히 지연이 발생하는 경우 지원 팀에 문의하세요.
</Accordion>

<Accordion title="Q25: 다른 이름(부모님의 이름 등)을 사용하여 가입하고 인증받을 수 있나요?">
  A: 아니요. 가입하는 사람은 사업의 법적 소유자와 모든 제출된 문서와 일치해야 합니다.

  부모님이 실제 사업 소유자인 경우:
  - 그들은 자신의 이메일과 휴대폰 번호를 사용하여 가입해야 합니다.
  - 모든 문서는 그들의 이름으로 되어 있어야 합니다.

  <Note>
  사업을 관리하는 동안 다른 사람의 이름을 사용하여 인증을 받으면 규정 준수 거부로 이어질 수 있습니다.
  </Note>
</Accordion>

<Accordion title="Q26: 인증 거부 사유를 확인하려면 어떻게 해야 하나요?">
  A: 다음에서 사유를 확인할 수 있습니다:
  - Dodo Payments 대시보드의 인증 섹션.
  - 이메일 알림(활성화된 경우)

  <Tip>
  보이지 않는 경우 compliance@dodopayments.com으로 문의하거나 등록된 이메일 ID를 팀원에게 DM하세요.
  </Tip>
</Accordion>

<Accordion title="Q27: 이전 계정이 거부된 경우 새 사업 계정을 만들 수 있나요?">
  A: 네 — 하지만 다음 조건을 충족해야 합니다:
  - 거부로 이어진 문제를 해결했습니다.
  - 위험 결정을 우회하려고 하지 않습니다(예: 사기 우려로 인한 탈퇴).

  <Note>
  근본적인 문제를 해결하지 않고 여러 계정을 생성하면 새 계정도 거부될 가능성이 높습니다.
  </Note>
</Accordion>

<Accordion title="Q28: 내 계정이 이전에 탈퇴되었다고 들었습니다. 이게 무슨 뜻인가요?">
  A: 귀하의 계정이 위험 팀에 의해 이전에 탈퇴되었다는 메시지가 표시되면:
  - Dodo Payments는 귀하의 활동이나 사업에 중대한 규정 준수 또는 위험 문제가 있다고 판단했습니다.
  - 동일한 신원이나 사업 세부정보를 사용하여 새 계정을 생성하려고 하면 다시 거부될 가능성이 높습니다.

  <Note>
  이것이 실수라고 생각되면 compliance@dodopayments.com으로 이메일을 보내 검토를 요청하세요. — 귀하의 사례를 뒷받침할 명확한 증거를 포함하세요.
  </Note>
</Accordion>

<Accordion title="Q29: 내 지급 인증을 더 빨리 통과시키려면 어떻게 해야 하나요?">
  A: 지연을 피하려면:
  - 명확하고 적절하게 형식화된 문서를 제출하세요.
  - Discord에서 반복적인 메시지/태그를 피하세요 — 이는 대기열을 느리게 합니다.
  - 지급 기준을 초과했거나 긴급한 일정이 있는 경우 언급하세요.
  - 요청 시 등록된 이메일 ID를 DM하세요.

  <Tip>
  대부분의 지급 검토는 제출에 문제가 없는 한 72 근무 시간 이내에 완료됩니다.
  </Tip>
</Accordion>

<Accordion title="Q30: 잘못된 양식(W-8BEN vs W-8BEN-E)을 선택한 경우 어떻게 해야 하나요?">
  A: 이러한 양식은 귀하의 법인 유형에 따라 다릅니다:
  - W-8BEN은 개인용입니다.
  - W-8BEN-E는 조직용입니다.

  잘못된 양식을 제출한 경우:
  - 지원 팀에 연락하여 양식을 재설정하세요.
  - 올바른 양식 유형을 작성하고 업로드하세요.
  - PAN이 올바른 필드에 있는지 확인하세요(개인의 경우 6a 라인).

  <Tip>
  혼란스러운 경우 IRS 공식 양식 가이드를 참조하세요.
  </Tip>
</Accordion>

<Accordion title="Q31: 나는 상호명이 있는 GST를 가지고 있습니다. 이것이 조직으로 간주되나요?">
  A: 아니요. 개인(개인 사업자)으로 GST에 등록된 경우 여전히 개인으로 간주됩니다.

  ✅ W-8BEN 사용
  ✅ 등록 시 "개인" 선택
  ❌ 등록된 법인이 아닌 경우 W-8BEN-E를 작성하거나 "조직"을 선택하지 마세요.
</Accordion>

<Accordion title="Q32: 내 계정이 탈퇴되었습니다. 새 계정을 만들 수 있나요?">
  A: 이전 탈퇴를 우회하기 위해 새 계정을 만드는 것은 허용되지 않습니다. 이를 시도할 경우:
  - 새 계정도 플래그가 지정되거나 거부될 수 있습니다.
  - 기본 사업체나 개인이 동일한 경우 위험 결정이 가입에 따라 이어집니다.

  <Tip>
  잘못 탈퇴되었다고 생각되면 compliance@dodopayments.com으로 이메일을 보내세요.
  </Tip>
</Accordion>

<Accordion title="Q33: Dodo Payments는 인도 SaaS 상인을 위해 GST를 어떻게 처리하나요?">
  A: Dodo Payments는 고객의 위치에 따라 인도 상인을 위해 **GST**를 자동으로 계산하고 적용합니다.
</Accordion>

<Accordion title="Q34: Dodo Payments는 국제 판매를 처리하나요?">
  A: 네, Dodo Payments는 국제 판매를 지원하며 모든 관련 규정을 관리합니다.
</Accordion>

<Accordion title="Q35: 도시, 주, 우편번호와 같은 주소 데이터 필드는 필수인가요? 고객에게 이러한 값을 요청하지 않을 수 있나요?">
  A: 네, 필수입니다. 우리는 기록의 상인으로서 송장을 생성하고 세금을 신고하며 모든 구매가 규정을 준수하도록 보장하기 때문에 고객의 청구 주소가 필요합니다. 고객의 세부정보를 미리 수집한 경우, 고객의 세부정보를 미리 채울 수 있는 API 기반 체크아웃 흐름을 구현할 수 있지만, 모든 구매에 대해 이러한 세부정보가 필요합니다.
</Accordion>

<Accordion title="Q36: 미국에서 판매하는 경우 세금을 신고해야 하나요?">
  A: 직접적으로는 아닙니다. Dodo Payments는 기록의 상인으로서 고객 측 세금을 처리하고 해당되는 경우 납부합니다. 여전히 다음을 수행해야 합니다:
  - 본국(예: 인도)에서 소득세 신고
  - 기본 30% 미국 원천징수를 피하기 위해 W-8BEN(-E) 제출
</Accordion>

<Accordion title="Q37: W-8BEN 또는 W-8BEN-E를 작성해야 할지 모르겠습니다. 도와주세요?">
  A: 필요한 양식을 결정하기 위해 다음 논리를 사용하세요:

  | 귀하가... | 제출할 양식 | 상인 유형 |
  |------------|----------------|---------------|
  | 개인 사업자, 프리랜서, 개인 | W-8BEN | 개인 |
  | Pvt Ltd, LLP, 등록된 조직 | W-8BEN-E | 조직 |

  여전히 확실하지 않으신가요? 다음 지표를 고려하세요:
  - **귀하의 은행 계좌가 개인 이름으로 되어 있다면**, 귀하는 개인일 가능성이 높습니다 → W-8BEN 사용
  - **귀하의 은행 계좌가 회사 이름으로 되어 있다면**, 귀하는 조직입니다 → W-8BEN-E 사용
  - **GST가 있지만 개인 은행 계좌로 운영하는 경우**, 여전히 개인으로 간주됩니다 → W-8BEN 사용

  <Tip>
  GST가 있는 개인 사업자라면 여전히 개인입니다. 등록된 법인(Pvt Ltd, LLP 등)만 W-8BEN-E를 사용해야 합니다.
  </Tip>
</Accordion>

<Accordion title="Q38: 결제를 받으려면 PAN이나 GST가 필요하나요?">
  A: 개인의 경우, PAN은 필수 요구 사항입니다(신원 확인 및 세금 목적). 등록된 사업체의 경우, 사업 PAN 또는 GSTIN을 제출할 수 있습니다. GST는 등록 기준을 초과하거나 입력 세액 공제를 원하지 않는 한 선택 사항입니다.
</Accordion>

<Accordion title="Q39: Udyam/MSME 인증서를 가지고 있습니다. W-8BEN 또는 W-8BEN-E를 제출해야 하나요?">
  A: Udyam/MSME 인증서를 가지고 있다고 해서 자동으로 조직이 되는 것은 아닙니다. 제출할 양식은 귀하의 은행 계좌 유형에 따라 다릅니다:

  - **개인 은행 계좌로 운영하는 경우** (Udyam/MSME 인증서가 있어도): **개인**으로 간주됩니다 → **W-8BEN** 제출
  - **회사 은행 계좌로 운영하는 경우** (등록된 법인): **조직**으로 간주됩니다 → **W-8BEN-E** 제출

  <Note>
  핵심 요소는 귀하의 은행 계좌가 개인 이름으로 되어 있는지 또는 회사 이름으로 되어 있는지입니다. 사업 등록 증명서가 있는지 여부는 중요하지 않습니다.
  </Note>
</Accordion>

<Accordion title="Q40: Dodo가 세금을 처리하는데 왜 W-8BEN(-E)를 제출해야 하나요?">
  A: Dodo는 고객의 판매세를 처리합니다. W-8BEN(-E)는 귀하에게 지급되는 금액에 대한 원천징수 세금을 위해 필요합니다.

  <Note>
  W-8BEN을 올바르게 제출하면 인도-미국 세금 조약에 따라 이중 과세를 피할 수 있습니다.
  </Note>
</Accordion>

## 결제 및 거래

<Accordion title="Q41: Dodo Payments로 어떤 종류의 결제를 수락할 수 있나요?">
  A: Dodo Payments는 **신용 카드**, **직불 카드**, **디지털 지갑** 등 다양한 결제 방법을 지원합니다. 또한 디지털 제품에 대한 일회성 결제 및 반복 구독 결제를 지원합니다.
</Accordion>

<Accordion title="Q42: 구독의 첫 번째 결제가 $0의 금액으로 성공으로 표시된 이유는 무엇인가요?">
  A: 제품에 시험 기간이 있는 경우, 첫 번째 결제는 자동으로 $0의 의무 결제가 됩니다. 이는 예상되는 동작이며 의무가 성공적으로 생성되었음을 확인합니다.
</Accordion>

<Accordion title="Q43: 시험 기간이 끝난 후 고객이 다시 수동으로 결제해야 하나요(제품 가격(예: $25))?">
  A: 아니요. 시험이 끝나면 시스템이 고객을 위해 자동으로 제품 가격(예: $25)을 생성합니다. 고객이 수동으로 결제할 필요는 없습니다.
</Accordion>

<Accordion title="Q44: 내 앱과 Dodo Payments 대시보드에서 시험 기간이 다르게 표시되면 어떻게 하나요?">
  A: 이는 일반적으로 귀하의 설정 또는 구현 문제를 나타냅니다. 먼저 테스트 모드에서 시험 + 구독 흐름을 테스트하고 작동하는지 확인한 후 라이브 모드로 전환하세요.
</Accordion>

<Accordion title="Q45: 라이브로 전환하기 전에 시험 기능을 테스트해야 하나요?">
  A: 네. 모든 것이 올바르게 작동하는지 확인하기 위해 시험 모드에서 기능을 완전히 구현하고 테스트하는 것이 좋습니다.
</Accordion>

<Accordion title="Q46: 결제 상태를 어떻게 추적할 수 있나요?">
  A: 대시보드의 **결제** 섹션에서 결제 상태를 추적할 수 있습니다. 각 거래는 상태 표시기와 함께 나열됩니다: **진행 중**, **실패**, 또는 **성공**.
</Accordion>

<Accordion title="Q47: 결제가 실패하면 어떻게 해야 하나요?">
  A: 결제가 실패하면 **결제 세부정보 페이지**에서 더 많은 정보를 확인하세요. 일반적인 이유로는 잘못된 결제 세부정보, 잔액 부족 또는 네트워크 문제가 있습니다. 고객에게 결제 링크를 다시 보내 결제를 재시도할 수 있습니다.
</Accordion>

<Accordion title="Q48: 사용자의 결제가 실패할 때 상인은 통지를 받나요?">
  A: 네. Dodo Payments는 사용자의 결제가 실패할 때 여러 채널을 통해 상인에게 통지합니다:
  - 이메일 알림
  - 웹훅 이벤트
  - 모바일 알림
  - 대시보드 알림
이로 인해 상인은 실패한 거래에 대한 정보를 유지할 수 있습니다.
</Accordion>

<Accordion title="Q49: 결제가 실패하면 구독은 어떻게 되나요?">
  A: 구독은 즉시 취소되지 않습니다. 대신 성공적인 결제가 이루어질 때까지 보류 상태로 전환됩니다.

  카드에 문제가 있는 경우, 다음 방법으로 결제 방법을 업데이트할 수 있습니다:
  - 대시보드에서 보류 중인 구독을 클릭하고 **결제 방법 업데이트** 버튼을 사용하세요.
  - 또는 API 엔드포인트를 사용하세요: https://docs.dodopayments.com/api-reference/subscriptions/update-payment-method
</Accordion>

<Accordion title="Q50: 내가 직접 결제를 재시도하면 실패한 결제 ID가 필요하나요?">
  A: 아니요. 새 결제 시도를 시작하려면 구독 ID만 필요합니다.
</Accordion>

<Accordion title="Q51: 이전 주기가 성공적이었는데도 왜 INCORRECT_NUMBER로 결제 실패가 발생하나요?">
  A: 구독 결제는 고객의 카드가 취소되었거나 교체되었거나 비활성화되었거나 이전 청구 주기 이후에 은행이나 결제 제공업체에 의해 플래그가 지정된 경우 INCORRECT_NUMBER로 실패할 수 있습니다. Dodo Payments는 결제 제공업체에서 반환된 오류 메시지를 단순히 표시합니다.
</Accordion>

<Accordion title="Q52: 내 사업이 환불 정책이 없더라도 차지백이 적용되나요?">
  A: 네. 귀하의 약관이나 웹사이트에 "환불 없음"이라고 명시하더라도 차지백은 카드 소지자의 은행 및 카드 네트워크 규칙(예: Visa, Mastercard)에 의해 주로 관리됩니다. "환불 없음" 정책은 카드 네트워크의 분쟁 해결 프로세스를 무시하지 않습니다. 고객이 분쟁을 제기하면 귀하의 사업은 여전히 차지백 조사에 따라야 합니다. 이는 귀하가 분쟁을 해결하거나 관리하기 위해 증거(배송 증명, 사용 로그, 약관 수락 등)를 제출해야 함을 의미합니다. 고객의 은행이 카드 소지자에게 유리하게 결정하면 차지백 수수료와 거래 취소가 귀하의 명시된 정책과 관계없이 적용됩니다. 우리는 명확한 커뮤니케이션을 유지하고 고객 사용을 추적하며 차지백이 발생할 때 설득력 있는 증거를 제공할 것을 권장합니다.
</Accordion>

<Accordion title="Q53: 왜 payment.status가 때때로 null인가요?">
  A: payment.status는 짧은 시간 동안 null일 수 있습니다:
  - 생성 직후 처리 완료 전에
  - 구독 청구 실패 또는 재시도 중
  결국, 상태는 succeeded, failed 또는 cancelled 중 하나가 됩니다.

  <Note>
  payment.succeeded 또는 payment.failed와 같은 웹훅 이벤트를 사용하여 더 신뢰할 수 있는 상태 추적을 수행하세요.
  </Note>
</Accordion>

<Accordion title="Q54: 왜 product_cart가 체크아웃 중에도 null인가요?">
  A:
  - 구독 결제의 경우, product_cart는 일회성 항목이 구매되지 않기 때문에 비어 있습니다.
  - product_cart는 일회성 결제에 대해서만 채워집니다.
</Accordion>

<Accordion title="Q55: 테스트 중에 내 실제 카드가 차단된 이유는 무엇인가요?">
  A: 아마도 테스트 모드가 아닌 라이브 모드에서 테스트했기 때문입니다.

  라이브 결제:
  - 실제 돈을 사용합니다.
  - 실제 은행을 통과합니다.
  - 거래량이 낮거나 일관성이 없으면 의심스러운 것으로 플래그가 지정될 수 있습니다.
  
  ✅ 항상 다음을 사용하여 테스트하세요:
  - 대시보드의 테스트 모드
  - 문서에 제공된 테스트 카드 번호
</Accordion>

<Accordion title="Q56: 내 고객이 결제를 완료했지만 웹훅을 받지 못했습니다. 왜 그런가요?">
  A: 체크리스트:
  - 대시보드 > 개발자 -> 웹훅에서 웹훅이 구성되어 있는지 확인하세요.
  - 웹훅이 비정상적인 2xx 응답으로 조용히 실패하고 있는지 확인하세요.
  - 로그를 확인하세요: 429 또는 타임아웃이 반환되었나요?
</Accordion>

<Accordion title="Q57: 내 대시보드에서는 성공으로 표시되지만 내 앱이 업데이트되지 않는 이유는 무엇인가요?">
  A: 아마도 결제 후 리디렉션 URL 쿼리 매개변수에만 의존하고 웹훅이나 API 확인 호출을 처리하지 않기 때문입니다.

  ✅ 수정:
  - 항상 payment.succeeded 웹훅을 수신하세요.
  - 또는 리디렉션 후 거래를 확인하기 위해 API를 쿼리하세요.
</Accordion>

<Accordion title="Q58: 결제 옵션으로 UPI가 보이지 않는데, 왜 그런가요?">
  A: 여러 가지 이유로 발생할 수 있습니다:
  - 청구 국가가 IN으로 설정되지 않았습니다.
  - 귀하의 사업에 대해 UPI가 활성화되지 않았습니다.
  
  비인도 상인인 경우, 고객에게 UPI가 나타나지 않습니다.
</Accordion>

<Accordion title="Q59: Dodo는 구독에 대해 UPI를 지원하나요?">
  A: 네, UPI 구독은 RBI 준수 의무와 함께 지원됩니다. UPI 구독은 RBI(인도 준비 은행) 규정에 따라 특정 요구 사항으로 운영됩니다:
  - **의무 한도**:
    - ₹15,000 미만의 요금: ₹15,000 INR의 요청 의무가 생성됩니다.
    - ₹15,000 이상의 요금: 정확한 구독 금액에 대한 의무가 생성됩니다.
  - **48시간 처리 지연**: 요금은 예정된 날짜에 시작되지만 실제 공제는 시작 후 48시간 후에 발생합니다(추가 2-3시간 연장될 수 있음).
  - **의무 취소**: 고객은 48시간 내에 은행 앱을 통해 의무를 취소할 수 있습니다.
  <Note>
  UPI 구독은 인도 카드 구독과 동일한 RBI 준수 의무 요구 사항을 가지고 있습니다. 자세한 내용은 [결제 방법 문서](/features/payment-methods#upi-and-rupay-india)를 참조하세요.
  </Note>
</Accordion>

<Accordion title="Q60: 정적 결제 링크에서 Google Pay / Apple Pay를 어떻게 활성화하나요?">
  A: Google Pay와 Apple Pay는 라이브 모드의 정적 결제 링크에서 자동으로 나타납니다. 이를 활성화하기 위해 추가 단계를 수행할 필요가 없습니다.
</Accordion>

<Accordion title="Q61: Dodo Payments에서 3DS를 어떻게 활성화하나요?">
  A: Dodo Payments 대시보드의 사업 설정에서 직접 3DS를 활성화할 수 있습니다. 사업 설정 → 결제 설정으로 이동하여 3DS 옵션을 켜세요.
</Accordion>

<Accordion title="Q62: API에서 전달하지 않고 청구 세부정보를 어떻게 수집하나요?">
  A: 사용자가 청구 정보를 직접 입력하도록 하려면:
  - 정적 링크를 사용하여 자동으로 청구 정보를 수집하세요.
  - 동적 흐름의 경우, 정보를 수동으로 수집하고 API에서 전달해야 합니다.
  
  동적 링크 생성 중에 청구를 수집하기 위한 내장 UI는 아직 없습니다.
</Accordion>

<Accordion title="Q63: 정적 결제 링크와 동적 결제 링크의 차이는 무엇인가요?">
  A: **정적 결제 링크**는 빠른 사용을 위한 코드 없는 옵션입니다. 이들은:
  - 대시보드에서 구성됩니다.
  - 자동으로 청구 정보를 수집합니다.
  - 고객/세션별로 사용자 정의할 수 없습니다.
  
  **동적 결제 링크**는 API를 통해 생성됩니다. 이들은:
  - 청구 정보를 명시적으로 전달해야 합니다.
  - 실시간으로 사용자 정의할 수 있습니다(가격, 사용자 ID, 리디렉션 URL 등).
  <Tip>
  코드를 작성하고 싶지 않다면 정적 링크를 사용하세요. 유연성이 필요하거나 사용자 정의 워크플로를 구축하고 싶다면 동적 링크를 사용하세요.
  </Tip>
</Accordion>

<Accordion title="Q64: 누군가 구독을 시작하면 두 개의 웹훅 이벤트를 받게 되나요?">
  A: 첫 번째 성공적인 구독 결제 시, 시험 기간이 없을 경우 일반적으로 다음 세 개의 웹훅을 받게 됩니다:
  - `subscription.created` 웹훅
  - `payment.succeeded` 웹훅
  - `subscription.renewal` 웹훅
</Accordion>

<Accordion title="Q65: subscription.cancelled_at는 무엇을 의미하나요? 예정된 취소인가요?">
  A: Dodo는 즉각적인 취소와 예정된 취소를 모두 지원합니다.

  취소 요청이 이루어지면:
  - **즉각적인 취소**: subscription.cancelled_at 타임스탬프가 즉시 설정되며 구독이 그 순간 종료됩니다.
  - **예정된 취소**: 현재 청구 기간이 끝날 때 취소하기 위해 구독 패치 API에서 `cancel_at_next_billing_date` 매개변수를 사용하세요.
</Accordion>

<Accordion title="Q66: 구독은 언제 청구되나요?">
  A: Dodo Payments는 제품에 시험 기간이 없는 경우 구독 생성 시 고객에게 즉시 청구합니다.

  <Note>
  인도 카드 및 UPI 구독의 경우, 반복 청구는 RBI 규정에 따라 48시간 처리 지연이 있습니다. 요금은 예정된 날짜에 시작되지만 고객의 계좌에서 실제 공제는 시작 후 48시간 후에 발생합니다. 이 48시간 창은 은행 API 응답에 따라 2-3시간 추가 연장될 수 있습니다.
  </Note>
</Accordion>

<Accordion title="Q67: Change Plan API가 빈 응답을 반환하는 이유는 무엇인가요?">
  A: Change Plan API는 응답 본문 없이 200 상태 코드만 반환하므로 응답이 비어 있는 것처럼 보일 수 있지만 계획 변경이 성공적으로 이루어졌습니다.
</Accordion>

<Accordion title="Q68: 월간 요금제에서 연간 요금제로 전환할 때 새 결제가 요청되지 않는 이유는 무엇인가요?">
  A: 요금제를 변경할 때 Dodo Payments는 저장된 카드를 자동으로 청구합니다. 청구가 실패하면 구독이 "보류 중"으로 전환됩니다. 따라서 수동 결제 흐름이 트리거되지 않습니다.
</Accordion>

<Accordion title="Q69: 내 제품 가격(₹100)이 체크아웃 중에 더 높은 금액(₹104)으로 표시되는 이유는 무엇인가요?">
  A: 이는 적응형 통화가 활성화되어 있을 때 발생합니다. 이 기능은 자동으로 추가 4% 요금을 부과합니다.
</Accordion>

<Accordion title="Q70: 체크아웃이 기존 활성 구독을 무시하고 새 구독을 생성하는 이유는 무엇인가요?">
  A: 체크아웃은 항상 새 구독을 생성합니다. 고객이 이미 활성 구독을 가지고 있는지 여부를 확인하지 않습니다. 귀하의 애플리케이션은 중복 청구를 피하기 위해 기존 구독자를 체크아웃 세션 시작을 차단해야 합니다.
</Accordion>

<Accordion title="Q71: 구독 업그레이드/다운그레이드 중 TRANSACTION_NOT_ALLOWED 오류가 발생하는 이유는 무엇인가요?">
  A: 이 오류는 Dodo Payments가 아닌 카드 발급사에서 발생합니다. 은행이 해당 카드에 대한 거래를 차단하고 있습니다.

  일반적인 이유:
  - 온라인, 국제 또는 반복 결제를 허용하지 않는 카드
  - 상인 카테고리(MCC)가 차단됨
  - 은행의 사기 규칙이 요금을 방지함
  - 선불 또는 제한된 카드는 이러한 결제를 지원하지 않을 수 있습니다.
</Accordion>

<Accordion title="Q72: 실제 결제를 하지 않고 라이브 모드에서 테스트할 수 있는 방법이 있나요?">
  A: 네. 100% 할인 코드를 생성하여 라이브 모드 체크아웃 흐름을 테스트할 수 있습니다. 이를 통해 실제 요금이 청구되지 않고 전체 결제 및 리디렉션 흐름을 완료할 수 있습니다.
</Accordion>

<Accordion title="Q73: $0 결제 방법이란 무엇인가요?">
  A: $0 결제는 총 지급 금액이 0이 되는 경우(예: 100% 할인을 적용하여) 발생합니다. 이러한 경우 실제 결제가 처리되지 않지만 체크아웃 및 구독 흐름은 여전히 정상적으로 작동합니다.
</Accordion>

<Accordion title="Q74: 내 계정이 정지된 것처럼 보이지만 규정 준수 팀은 모든 것이 괜찮다고 합니다. 어떻게 해야 하나요?">
  A: 대부분의 경우, 이는 귀하가 실수로 오래되거나 비활성 사업 계정으로 전환했기 때문입니다. Dodo Payments 대시보드에서 활성 사업 계정으로 다시 전환하세요. 전환 후에도 문제가 지속되면 Dodo Payments 지원 팀에 문의하여 도움을 받으세요.
</Accordion>

## 지급 및 은행 통합

<Accordion title="Q75: Dodo Payments와 함께 지급은 어떻게 작동하나요?">
  A: 지급은 귀하가 수집한 판매 수익을 연결된 은행 계좌로 이체하는 것입니다. Dodo Payments는 여러 지급 주기를 지원합니다:

  - **격주(기본)**:
    - 청구 기간 1: 1일부터 15일까지 → 같은 달 18일 지급
    - 청구 기간 2: 16일부터 월말까지 → 다음 달 4일 지급
  - **주간**: 거래량이 높은 사업체에 대해 사용 가능(요청 시 지원 팀에 문의)
  - **월간**:
    - 청구 기간: 1일부터 월말까지 → 다음 달 11일 지급
  지급은 모든 인증이 성공적으로 완료되고 지급 기준이 충족된 경우에만 시작됩니다. 지급이 시작된 후 자금이 귀하의 은행 계좌에 나타나는 데 일반적으로 1-2 근무 일이 소요됩니다.

  <Note>
  지급 처리 날짜가 주말이나 은행 공휴일에 해당하는 경우 지연이 발생할 수 있습니다. 모든 사업체의 기본 지급 주기는 격주입니다.
  </Note>
</Accordion>

<Accordion title="Q76: 수수료 및 세금 공제 후 지급을 처리하기 위한 최소 기준 금액은 얼마인가요?">
  A: 최소 지급 기준은 **$50 USD**(또는 ₹1,000 INR)입니다. 플랫폼 수수료 및 세금 공제 후입니다. 대시보드에서 USD 지갑 기준을 **≥ $50**으로 사용자 정의할 수 있습니다. 잔액이 기준을 충족하지 않으면 다음 지급 주기로 이월됩니다.

  정확한 지갑 금액을 확인하고 지급 기준을 관리하려면 대시보드의 [지급 섹션](https://app.dodopayments.com/business/payouts)을 확인하세요.

  <Note>
  유연한 기준 기능은 현재 USD/EUR/GBP 지갑에만 제공됩니다. INR 지갑 기준은 ₹1,000으로 고정되어 있습니다.
  </Note>
</Accordion>

<Accordion title="Q77: 지급을 받기 위해 내 은행 계좌를 어떻게 연결하나요?">
  A: 은행 계좌를 연결하려면 대시보드의 **지급 설정** 섹션으로 이동하여 **은행 계좌 추가**를 클릭하세요. 은행 세부정보를 입력하고 계좌를 확인해야 합니다.
</Accordion>

<Accordion title="Q78: 내 지급 상태를 추적할 수 있나요?">
  A: 네, 대시보드의 **지급** 섹션에서 지급 상태를 추적할 수 있습니다.
</Accordion>

<Accordion title="Q79: 지급 수수료는 무엇이며 얼마나 되나요?">
  A: 지급 수수료는 고객으로부터 수집한 돈을 귀하의 은행 계좌로 이체하는 거래 비용을 의미합니다. 우리는 귀하의 국가에서 현지 파트너와 협력하여 지급 수수료를 줄이기 위해 지속적으로 개선하고 있습니다. 자세한 내용은 [가격 및 수수료 구조](https://dodopayments.com/pricing)를 참조하세요.
</Accordion>

<Accordion title="Q80: '지급 처리됨' 이메일을 받았습니다. 자금은 언제 도착하나요?">
  A: 지급은 일반적으로 1-2 근무일 이내에 귀하의 은행 계좌에 반영됩니다.

  <Note>
  주말이나 은행 공휴일에는 이체가 완료되지 않습니다. 지급이 금요일에 처리되었다면 월요일이나 화요일까지 기다려야 합니다.
  </Note>
</Accordion>

<Accordion title="Q81: Payoneer를 지급 계좌로 추가하려면 어떻게 하나요?">
  A: 대시보드의 은행 정보 양식에 Payoneer 은행 세부정보를 입력하기만 하면 됩니다. 별도의 Payoneer 통합은 아직 없습니다.
</Accordion>

<Accordion title="Q82: Payoneer 계좌로 지급을 받을 수 있나요?">
  A: 네. Payoneer 계좌로 지급을 받을 수 있습니다:
  - Dodo 대시보드의 은행 정보 양식에 Payoneer 계좌 세부정보를 입력하세요.
  - 귀하의 계좌가 승인되었고 USD 또는 인출하려는 통화를 지원하는지 확인하세요.
  별도의 Payoneer 통합은 없으며 일반 은행 계좌로 처리됩니다.
</Accordion>

<Accordion title="Q83: 내 은행 정보와 규정 준수 양식이 아직 검토 중입니다. 어떻게 해야 하나요?">
  A: 그냥 기다리세요. 우리의 규정 준수 팀은 일반적으로 양식을 1-2 근무일 이내에 검토합니다. 긴급한 경우:
  - support@dodopayments.com으로 이메일을 보내세요.
</Accordion>

<Accordion title="Q84: 내 지급일이 금요일이나 주말에 해당하면 어떻게 되나요?">
  A: 지급일이 금요일인 경우 일반적으로 같은 날 처리됩니다. 그러나:
  - 은행 공휴일이나 주말이 있는 경우 다음 근무일(일반적으로 월요일)로 지연될 수 있습니다.
  - 정확한 시간은 귀하의 은행 처리 규칙에 따라 달라질 수 있습니다.
  <Note>
  의심스러운 경우 대시보드를 확인하거나 등록된 이메일로 지원 팀에 문의하세요.
  </Note>
</Accordion>

<Accordion title="Q85: 긴급 출시나 광고 캠페인이 계획되어 있는 경우 지급 규정 준수를 어떻게 신속하게 진행할 수 있나요?">
  A: 모든 것을 제출하고 지급 규정 준수를 기다리고 있다면:
  - 지원 채팅에서 긴급성을 명확히 언급하세요(예: 라이브 출시, 유료 캠페인).
  - 사업 ID와 이메일을 포함하세요.
  - 스팸을 보내거나 여러 팀원을 태그하지 마세요 — 이는 대기열을 느리게 할 수 있습니다.
</Accordion>

<Accordion title="Q86: 잘못된 은행 계좌 세부정보를 제출했습니다. 어떻게 업데이트하나요?">
  A: 잘못된 은행 세부정보를 입력한 경우:
  - Intercom 또는 Discord 또는 이메일을 통해 지원 팀에 문의하세요.
  - Discord를 통해 연락하는 경우 이메일 ID를 DM하고 은행 검증 섹션의 수동 재설정을 요청하세요.
  - 재제출을 위해 섹션을 다시 열도록 규정 준수 팀을 기다리세요.
  <Note>
  제출 후 대시보드에서 직접 은행 세부정보를 편집하는 것은 허용되지 않을 수 있습니다.
  </Note>
</Accordion>

<Accordion title="Q87: 등록된 회사가 있는 경우 개인 은행 계좌를 사용할 수 있나요?">
  A: 아니요. 회사 또는 법인으로 등록된 경우 해당 법인의 은행 계좌를 사용해야 합니다.
</Accordion>

<Accordion title="Q88: 내 나라에서 SWIFT 코드를 사용하지 않으면 내 인증에 어떤 영향을 미치나요?">
  A: 귀하의 나라에서 SWIFT 코드를 발급하지 않는 경우, 이는 귀하의 인증을 차단하지 않지만, 우리의 규정 준수 팀에서 수동 검토가 필요합니다.

## 송장 및 보고서

<Accordion title="Q90: Dodo Payments는 송장을 자동으로 생성하나요?">

<Accordion title="Q91: 송장을 사용자 정의할 수 있나요?">

</Accordion>

<Accordion title="Q92: 송장에 접근하거나 다운로드하려면 어떻게 하나요?">

</Accordion>

<Accordion title="Q93: 내 거래 및 지급에 대한 보고서를 보려면 어떻게 해야 하나요?">
  A: 대시보드의 **보고서** 섹션에서 자세한 보고서를 생성할 수 있습니다. 이 보고서에는 거래, 환불 및 지급에 대한 데이터가 포함되어 있으며, 날짜별로 필터링하고 CSV 형식으로 내보낼 수 있습니다.
</Accordion>

<Accordion title="Q94: 환불을 어떻게 발행하나요?">

</Accordion>

<Accordion title="Q96: 분쟁이나 차지백에 직면했습니다. 어떻게 해야 하나요?">

  - 제품 배송 증명, 사용 로그 또는 고객 커뮤니케이션의 증거를 제출할 수 있습니다.
  - 그러나 최종 결정은 고객의 발급 은행에 달려 있습니다.
  <Note>
  support@dodopayments.com으로 다음을 포함하여 이메일을 보내세요:
  - 분쟁 ID
  - 결제 ID
  - 관련 증거
  </Note>

<Accordion title="Q97: 고객이 서비스를 사용한 후 요금을 분쟁했습니다. 이의 제기를 할 수 있나요?">

## 웹훅 및 통합

<Accordion title="Q98: Dodo Payments 웹훅을 Python에서 수동으로 검증할 수 있나요?">
  A: 네. 수동 검증이 가능하지만 올바른 서명 단계를 엄격히 따라야 합니다. 더 쉽고 신뢰할 수 있는 접근 방식을 위해 Dodo Payments는 standardwebhooks 패키지를 사용하는 것을 권장합니다:
  

  ```python
  from standardwebhooks.webhooks import Webhook
  
  wh = Webhook(webhook_secret_key)
  payload = request.get_data(as_text=True)
  headers = request.headers
  
  if not wh.verify(payload, headers):
      return jsonify({'error': 'Invalid webhook signature'}), 401
 
  Then parse the payload as needed.
  ```

  </Accordion>

<Accordion title="Q99: 수동 웹훅 검증에 대한 공식 문서를 어디에서 찾을 수 있나요?">
A: 공식 Dodo Payments 문서는 여기에서 참조할 수 있습니다: https://docs.dodopayments.com/developer-resources/webhooks#manual-verification-alternative
</Accordion>

<Accordion title="Q100: 내 웹훅이 POST 요청을 받지 않고 GET 요청만 받는 이유는 무엇인가요?">
A: 리디렉션 URL과 웹훅 URL을 혼동하고 있을 가능성이 높습니다. 리디렉션 URL은 결제 후 사용자가 전송되는 곳이며 GET 요청에서 `?subscription_id=...&status=active`를 포함할 수 있습니다. 이는 귀하의 웹훅이 아닙니다.

Dodo Payments에서 구조화된 POST 페이로드를 수신하려면:
- 대시보드 → 개발자 → 웹훅에서 웹훅을 설정하세요.
- URL이 POST 요청을 수락하고 서명을 검증하는지 확인하세요.

<Note>
웹훅과 리디렉션 URL을 모두 가질 수 있습니다. 이들은 서로 다른 목적을 수행합니다.
</Note>
</Accordion>

<Accordion title="Q101: Python SDK를 사용할 때 401 Unauthorized 오류가 발생하는 이유는 무엇인가요?">
A: DodoPayments 클라이언트를 초기화할 때 올바른 환경을 전달하지 않았을 가능성이 높습니다.

<Check>
**수정:**

```python
client = DodoPayments(
bearer_token="your_api_key",
      environment="live_mode"  # or "test_mode"
)
```

</Check>
</Accordion>

<Accordion title="Q102: 테스트를 위해 웹훅 이벤트를 수동으로 트리거할 수 있는 방법이 있나요?">
A: 웹훅 대시보드를 통해 가능합니다. 대시보드 -> 개발자 -> 웹훅으로 이동하세요.
</Accordion>

<Accordion title="Q103: 웹훅과 리디렉션 URL의 차이는 무엇인가요?">
A:
- **웹훅 URL**: 대시보드에서 구성됩니다. 이벤트에 대한 POST 페이로드를 수신합니다.
- **리디렉션 URL**: 결제 API 요청에서 전송됩니다. 쿼리 매개변수(`?status=success`)와 함께 사용자를 귀하의 앱으로 다시 보냅니다.
<Tip>
웹훅을 사용하여 DB 업데이트와 같은 작업을 자동화하고 리디렉션을 사용하여 감사 화면을 표시하세요.
</Tip>
</Accordion>

<Accordion title="Q104: 왜 구독 제품에 대한 payment.succeeded 웹훅에서 제품 ID를 받지 못했나요?">
A: 구독 제품에 대한 payment.succeeded 웹훅 이벤트는 product_id를 포함하지 않습니다. 대신 구독과 관련된 `subscription_id`와 기타 관련 세부정보를 제공합니다.
</Accordion>

<Accordion title="Q105: 내가 받는 웹훅이 실제로 Dodo Payments에서 온 것인지 악의적인 행위자가 아닌지 어떻게 확인하나요?">
A: 수신하는 웹훅이 실제로 Dodo Payments에서 온 것인지 확인하려면 다음 단계를 따르세요:
1. **서명 확인**: Dodo Payments에서 전송하는 각 웹훅에는 **`webhook-signature`** 헤더에 고유한 서명이 포함되어 있습니다. 이 서명은 귀하의 웹훅 비밀 키와 이벤트의 페이로드를 사용하여 생성됩니다.
2. **비밀 키 검색**: 웹훅의 비밀 키를 확보하세요. 이는 Dodo Payments 대시보드의 웹훅 설정에서 찾을 수 있습니다.
3. **서명 재계산**: Dodo Payments와 동일한 서명 알고리즘(HMAC-SHA256)을 사용하여 서버에서 서명을 계산하세요. 웹훅의 페이로드와 웹훅 비밀 키를 사용하세요.
4. **서명 비교**: 재계산된 서명을 **`webhook-signature`** 헤더에 제공된 서명과 비교하세요. 일치하면 웹훅이 진짜입니다. 일치하지 않으면 거부해야 합니다.
5. **HTTPS 사용**: 웹훅 엔드포인트가 HTTPS를 사용하여 통신을 보호하고 데이터 전송 중 변조를 방지하는지 확인하세요.
이 단계를 구현하면 Dodo Payments에서 전송된 웹훅의 진위를 자신 있게 확인할 수 있습니다.

<Tip>우리는 서명을 검증하기 위해 StandardWebhooks에서 제공하는 패키지를 사용하는 것을 권장하며, 직접 구성하지 마세요.</Tip>
</Accordion>

<Accordion title="Q106: 왜 첫 달 결제에도 subscription.renew 웹훅을 받나요?">
A: 구독의 경우, `subscription.renew`는 구독 금액이 공제될 때마다 트리거됩니다. 따라서 이는 예상되는 동작입니다.
</Accordion>

<Accordion title="Q107: 웹훅을 로컬에서 테스트하려면 어떻게 해야 하나요?">
A: 웹훅을 로컬에서 테스트하려면 다음 단계를 따르세요:

1. **터널링 도구 사용**: [ngrok](https://ngrok.com/), [LocalTunnel](https://localtunnel.github.io/www/) 또는 [Cloudflare Tunnel](https://developers.cloudflare.com/cloudflare-one/connections/connect-apps/)과 같은 도구를 사용하여 로컬 서버를 인터넷에 노출할 수 있습니다. 이를 통해 Dodo Payments가 로컬 환경으로 웹훅 이벤트를 전송할 수 있는 공개 URL을 제공합니다.
2. **웹훅 엔드포인트 설정**: 로컬 애플리케이션에서 수신 웹훅 이벤트를 처리할 웹훅 엔드포인트를 생성하세요. HTTP POST 요청을 수신하도록 설정하세요.
3. **Dodo Payments에서 웹훅 구성**: Dodo Payments 대시보드로 이동하여 웹훅 설정에서 웹훅 URL을 터널링 도구에서 생성된 공개 URL로 업데이트하세요.
</Accordion>

<Accordion title="Q108: 결제 후 내 웹훅이 호출되지 않는 이유는 무엇인가요?">
A: 다음을 확인하세요:
- 웹훅 URL과 리디렉션 URL을 혼동하지 않았는지
- 웹훅이 개발자 > 웹훅에서 구성되어 있는지
- 귀하의 URL이 POST 요청을 수락하고 2xx로 응답하는지
- 서명 검증(활성화된 경우)이 올바르게 구현되었는지

GET 요청과 함께 ?status=success가 표시되면 리디렉션 URL을 디버깅하고 있는 것입니다. 웹훅이 아닙니다.
</Accordion>

<Accordion title="Q109: URL 쿼리 매개변수에서 status=success에 의존하는 것이 안전한가요?">
A: 아니요. 빠른 해결책이지만 URL 쿼리 매개변수는 변조될 수 있습니다.

URL 쿼리 매개변수로 redirect_url?status=success를 사용하는 경우:
- API 호출로 거래를 이중 확인하세요.
- 또는 payment.succeeded 웹훅을 기다리세요.
<Tip>
리디렉션을 사용자 확인으로 간주하고 결제 증명으로 간주하지 마세요.
</Tip>
</Accordion>

<Accordion title="Q110: Dodo Payments를 기존 웹사이트나 플랫폼과 통합할 수 있나요?">
A: 네, Dodo Payments는 웹사이트나 플랫폼과의 원활한 통합을 위한 **API 통합** 및 **SDK 지원**을 제공합니다. API 문서, SDK 가이드 및 통합 리소스는 대시보드의 **개발자 섹션**에서 찾을 수 있습니다.
</Accordion>

모든 화면에 "테스트 모드" 레이블이 표시됩니다. 전환할 때 해당 API 키 또는 환경을 사용해야 합니다.

이렇게 하면 재정적 영향 없이 엔드 투 엔드 흐름을 디버깅할 수 있습니다.

<Accordion title="Q109: Is it safe to rely on status=success in URL query params?">
A: No. While it's a quick win, URL query params can be tampered with.

If you're using redirect_url?status=success:
- Double-validate the transaction with an API call
- Or wait for payment.succeeded webhook

<Tip>
Treat redirects as user-facing confirmations, not proof of payment.
</Tip>
</Accordion>

## 결제 실패

<Accordion title="Q125: 내 결제 상태가 '시작되지 않음'으로 표시되는 이유는 무엇인가요?">
  A: 이는 결제 링크가 생성되었지만 사용자가 결제 페이지를 열거나 결제를 시도하지 않았음을 의미합니다.
</Accordion>

<Accordion title="Q126: 내 결제 상태가 '결제 방법 필요'로 표시되는 이유는 무엇인가요?">
  A: 이는 사용자가 결제 페이지를 열고 고객 세부 정보를 입력했지만 카드 또는 결제 세부 정보를 추가하기 전에 종료했음을 의미합니다.
</Accordion>

<Accordion title="Q127: '유효하지 않은 인증 세부정보' 오류가 발생하는 이유는 무엇인가요?">
  A: 이는 일반적으로 3D Secure (3DS) 인증이 실패하거나 사용자가 완료하지 않았을 때 발생합니다.
</Accordion>

<Accordion title="Q128: 결제에 대해 '인증 실패' 오류가 발생하는 이유는 무엇인가요?">
  A: 인증 오류로 인해 결제가 실패했습니다. 이 문제에 대해 은행에 확인해 주시기 바랍니다. 안타깝게도 저희 측에서는 더 이상 할 수 있는 일이 없습니다. 다른 카드나 다른 결제 방법을 시도해 보시기를 권장합니다.
</Accordion>
