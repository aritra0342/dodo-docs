## अवलोकन

ऑन-डिमांड सब्सक्रिप्शन आपको एक ग्राहक के भुगतान विधि को एक बार अधिकृत करने और फिर जब भी आवश्यकता हो, परिवर्तनीय राशि चार्ज करने की अनुमति देता है, बजाय इसके कि एक निश्चित कार्यक्रम पर। यह सुविधा सभी खातों के लिए उपलब्ध है—कोई अनुमोदन आवश्यक नहीं।

इस गाइड का उपयोग करें:
- एक ऑन-डिमांड सब्सक्रिप्शन बनाएं (वैकल्पिक प्रारंभिक मूल्य के साथ एक जनादेश अधिकृत करें)
- कस्टम राशियों के साथ बाद के चार्ज को ट्रिगर करें
- वेबहुक का उपयोग करके परिणामों को ट्रैक करें

सामान्य सब्सक्रिप्शन सेटअप के लिए, [सब्सक्रिप्शन इंटीग्रेशन गाइड](/developer-resources/subscription-integration-guide) देखें।

## पूर्वापेक्षाएँ

- डोडो पेमेंट्स व्यापारी खाता और API कुंजी
- वेबहुक गुप्त कुंजी कॉन्फ़िगर की गई और घटनाओं को प्राप्त करने के लिए एक एंडपॉइंट
- आपके कैटलॉग में एक सब्सक्रिप्शन उत्पाद

<Tip>
यदि आप ग्राहक से होस्टेड चेकआउट के माध्यम से आदेश को स्वीकृत करवाना चाहते हैं, तो `payment_link: true` सेट करें और `return_url` प्रदान करें।
</Tip>

## ऑन-डिमांड कैसे काम करता है

1. आप `on_demand` ऑब्जेक्ट के साथ एक subscription बनाते हैं ताकि एक भुगतान विधि को अधिकृत किया जा सके और आवश्यकतानुसार प्रारंभिक शुल्क एकत्रित किया जा सके।
2. बाद में, आप समर्पित charge endpoint का उपयोग करके उस subscription के खिलाफ कस्टम राशि के साथ शुल्क बनाते हैं।
3. आप अपने सिस्टम को अपडेट करने के लिए webhooks (जैसे `payment.succeeded`, `payment.failed`) को सुनते हैं।

## एक ऑन-डिमांड सब्सक्रिप्शन बनाएं

एंडपॉइंट: [POST /checkouts](/api-reference/checkout-sessions/create)

मुख्य अनुरोध फ़ील्ड (शरीर):  
कृपया उन्हें [चेकआउट सत्र बनाएं](/api-reference/checkout-sessions/create) में देखें।

```javascript
import DodoPayments from 'dodopayments';

const client = new DodoPayments({
  bearerToken: process.env.DODO_PAYMENTS_API_KEY,
  environment: 'test_mode', // defaults to 'live_mode'

async function main() {
  const subscription = await client.checkoutSessions.create({

    customer: { customer_id: 'cus_123' },

        on_demand: {
            mandate_only: true // set false to collect an initial charge
            // product_price: 1000, // optional: charge $10.00 now if mandate_only is false
            // product_currency: 'USD',
            // product_description: 'Custom initial charge',
            // adaptive_currency_fees_inclusive: false,
        }
    }
  });

  console.log(subscription.checkout_url);
}

main().catch(console.error);
```

```go
        SubscriptionData: &dodo.SubscriptionData{
            OnDemand: &dodo.OnDemandSubscription{

                // ProductDescription: "Custom initial charge",
                // AdaptiveCurrencyFeesInclusive: false,

    }

}
```

## भुगतान पुनः प्रयास

हमारी धोखाधड़ी पहचान प्रणाली आक्रामक पुनः प्रयास पैटर्न को अवरुद्ध कर सकती है (और उन्हें संभावित कार्ड परीक्षण के रूप में चिह्नित कर सकती है)। एक सुरक्षित पुनः प्रयास नीति का पालन करें।

<Warning>
Burst retry पैटर्न को हमारे जोखिम सिस्टम्स और प्रोसेसर धोखाधड़ी या संदिग्ध कार्ड परीक्षण के रूप में चिह्नित कर सकते हैं। क्लस्टर्ड retries से बचें; नीचे दिए गए backoff कार्यक्रम और समय-संरेखण निर्देशों का पालन करें।
</Warning>

### सुरक्षित पुनः प्रयास नीतियों के लिए सिद्धांत

- **Backoff mechanism**: retries के बीच exponential backoff का उपयोग करें।
- **Retry limits**: कुल retries को सीमित करें (अधिकतम 3–4 प्रयास)।
- **Intelligent filtering**: केवल retryable failures (जैसे नेटवर्क/issuer त्रुटियाँ, अपर्याप्त धन) पर ही पुनः प्रयास करें; हार्ड declines पर कभी पुनः प्रयास न करें।
- **Card testing prevention**: `DO_NOT_HONOR`, `STOLEN_CARD`, `LOST_CARD`, `PICKUP_CARD`, `FRAUDULENT`, `AUTHENTICATION_FAILURE` जैसी विफलों को पुनः प्रयास न करें।
- **Vary metadata (optional)**: यदि आप अपना retry सिस्टम बनाए रखते हैं, तो metadata (जैसे `retry_attempt`) के माध्यम से retries को अलग करें।

### सुझाई गई पुनः प्रयास अनुसूची (सब्सक्रिप्शन)

- **1st प्रयास**: जब आप चार्ज बनाते हैं तो तुरंत
- **2nd प्रयास**: 3 दिन बाद
- **3rd प्रयास**: 7 और दिनों के बाद (कुल 10 दिन)
- **4th प्रयास (अंतिम)**: 7 और दिनों के बाद (कुल 17 दिन)

अंतिम चरण: यदि अभी भी भुगतान नहीं हुआ है, तो अपनी नीति के आधार पर सब्सक्रिप्शन को अनपेक्षित के रूप में चिह्नित करें या इसे रद्द करें। ग्राहक को उनके भुगतान विधि को अपडेट करने के लिए सूचित करें।

### बर्स्ट पुनः प्रयासों से बचें; प्राधिकरण समय के अनुसार संरेखित करें

- अपने पोर्टफोलियो में "burst" व्यवहार से बचने के लिए retries को मूल authorization timestamp से जोड़ें।
- उदाहरण: यदि ग्राहक आज 1:10 बजे trial या mandate शुरू करता है, तो अपने backoff के अनुसार अगले दिनों (जैसे +3 दिन → 1:10 बजे, +7 दिन → 1:10 बजे) पर follow-up retries निर्धारित करें।
- वैकल्पिक रूप से, यदि आप अंतिम सफल भुगतान समय `T` संग्रहीत करते हैं, तो दिन के समय-संरेखण को बनाए रखने के लिए अगले प्रयास को `T + X days` पर निर्धारित करें।

<Note>
Time-zone और DST: शेड्यूलिंग के लिए एक सुसंगत समय मानक का उपयोग करें और केवल प्रदर्शन के लिए रूपांतरण करें ताकि अंतराल बने रहें।
</Note>

### अस्वीकृति कोड जिन पर आपको पुनः प्रयास नहीं करना चाहिए

- `STOLEN_CARD`
- `DO_NOT_HONOR`
- `FRAUDULENT`
- `PICKUP_CARD`
- `AUTHENTICATION_FAILURE`
- `LOST_CARD`

<Note>
डिक्लाइन कारणों की विस्तृत सूची और क्या वे उपयोगकर्ता द्वारा ठीक किए जा सकते हैं, यह जानने के लिए [Transaction Failures](/api-reference/transaction-failures) दस्तावेज़ देखें।
</Note>

<Tip>
केवल सॉफ्ट/अल्पकालिक समस्याओं (उदा., `insufficient_funds`, `issuer_unavailable`, `processing_error`, नेटवर्क टाइमआउट) पर पुनः प्रयास करें। यदि वही decline दोबारा हो रहा है, तो आगे के retries को रोक दें।
</Tip>

### कार्यान्वयन दिशानिर्देश (कोई कोड नहीं)

- एक scheduler/queue का उपयोग करें जो सटीक timestamps को बनाए रखता है; अगले प्रयास की गणना दिन के समय के उसी ऑफ़सेट पर करें (उदा., `T + 3 days` को उसी HH:MM पर)।
- अगले प्रयास की गणना करने के लिए अंतिम सफल भुगतान timestamp `T` को बनाए रखें और संदर्भित करें; कई subscriptions को एक ही समय पर न जमा करें।
- हमेशा अंतिम decline कारण का मूल्यांकन करें; ऊपर दी गई skip सूची में शामिल हार्ड declines के लिए retries रोक दें।
- आकस्मिक उछाल को रोकने के लिए प्रति ग्राहक और प्रति खाते concurrent retries की सीमा निर्धारित करें।
- सक्रिय रूप से संवाद करें: अगले निर्धारित प्रयास से पहले ग्राहक को ईमेल/SMS भेजें ताकि वे अपनी भुगतान विधि अपडेट कर सकें।
- metadata को केवल observability के लिए उपयोग करें (उदा., `retry_attempt`); तुच्छ क्षेत्रों को घुमाकर धोखाधड़ी/जोखिम प्रणालियों से बचने की कोशिश कभी न करें।

## वेबहुक के साथ परिणामों को ट्रैक करें

ग्राहक यात्रा को ट्रैक करने के लिए वेबहुक हैंडलिंग लागू करें। [वेबहुक लागू करना](/developer-resources/integration-guide#implementing-webhooks) देखें।

- **subscription.active**: जनादेश अधिकृत और सब्सक्रिप्शन सक्रिय
- **subscription.failed**: निर्माण विफल (जैसे, जनादेश विफलता)
- **subscription.on_hold**: सब्सक्रिप्शन होल्ड पर रखा गया (जैसे, अनपेक्षित स्थिति)
- **payment.succeeded**: चार्ज सफल
- **payment.failed**: चार्ज विफल

<Tip>
On-demand फ्लो के लिए, usage-based शुल्कों का मिलान करने के लिए `payment.succeeded` और `payment.failed` पर ध्यान दें।
</Tip>

## समस्या निवारण

- **422 Invalid Request**: सुनिश्चित करें कि `on_demand.mandate_only` निर्माण के समय प्रदान किया गया है और `product_price` शुल्कों के लिए प्रदान किया गया है।
- **Currency errors**: यदि आप `product_currency` को ओवरराइड करते हैं, तो पुष्टि करें कि यह आपके खाते और ग्राहक के लिए समर्थित है।
- **No webhooks received**: अपने webhook URL और signature secret विन्यास की पुष्टि करें।
