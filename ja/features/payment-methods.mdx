Dodo Paymentsは、世界中の顧客にサービスを提供するために、幅広い支払い方法をサポートしています。すべてのリストされた支払い方法は、マーチャントアカウントが確認され、ライブモードが有効になると自動的に利用可能になります。

<Info>
**自動アクティベーション:** あなたの側で追加の設定は必要ありません。すべての支払い方法はDodo Paymentsによって事前に設定され、管理されています。オプションで、以下に説明するAPI設定を使用して、チェックアウト時に表示される方法を制限することができます。
</Info>

## 支払い方法の設定

すべての支払い方法は、顧客の所在地と通貨に基づいて自動的に利用可能ですが、チェックアウト時に表示される支払いオプションを制限するために、`allowed_payment_method_types`パラメータをオプションで使用できます。

### 支払い方法を設定するタイミング

許可された支払い方法を設定する理由は次のとおりです：

- ビジネス要件に基づいて顧客に表示される支払いオプションを制御する
- 特定の製品タイプや取引フローに対して支払い方法を制限する
- 異なる支払い体験全体で一貫性を維持する
- ターゲットオーディエンスに最適なチェックアウト体験を提供する

### 実装

<Steps>
<Step title="APIリクエストにパラメータを追加">
支払いまたはサブスクリプション作成リクエストに`allowed_payment_method_types`フィールドを含めます：

```json
{
  "allowed_payment_method_types": [
    "credit",
    "debit",
    "apple_pay"
  ]
}
```

</Step>

<Step title="設定をテスト">
指定された支払い方法のみがチェックアウトに表示されることを確認します。

<Tip>
異なる顧客の所在地と通貨で支払い方法の可用性をテストして、設定が期待通りに機能することを確認してください。
</Tip>
</Step>
</Steps>

### ベストプラクティス

<AccordionGroup>
<Accordion title="デフォルトの動作">
`allowed_payment_method_types`が指定されていない場合、すべての適格な支払い方法は、顧客の所在地、通貨、および取引タイプに基づいて利用可能になります。このパラメータは、支払い方法を制限する必要がある場合にのみ使用してください。
</Accordion>

<Accordion title="地域の考慮事項">
支払い方法を選択する際には、ターゲット市場を考慮してください。地域特有のオプション（インドのUPIやオランダのiDEALなど）を含めることで、これらの市場でのコンバージョン率を大幅に向上させることができます。
</Accordion>

<Accordion title="監視と最適化">
顧客が最も頻繁に使用する支払い方法を追跡し、実際の使用パターンやコンバージョンデータに基づいて許可された方法を調整します。
</Accordion>
</AccordionGroup>

### 重要な考慮事項

支払い方法の可用性は、複数の要因に依存します：
- **顧客の所在地**: 一部の支払い方法は地域特有です
- **マーチャント設定**: アカウントの設定と機能
- **通貨**: サポートされている支払い方法は通貨によって異なります
- **取引タイプ**: 一部の方法はサブスクリプションをサポートしていないか、最低金額があります

<Note>
`allowed_payment_method_types`に支払い方法を追加しても、それが表示されることは保証されません - これらの要因に基づいても適格である必要があります。
</Note>

## トラブルシューティング

<AccordionGroup>
<Accordion title="チェックアウトに支払い方法が表示されない">
**考えられる原因:**
- 支払い方法が`allowed_payment_method_types`に含まれていない
- 支払い方法が顧客の所在地でサポートされていない
- 支払い方法が取引通貨をサポートしていない
- サブスクリプションの場合: 支払い方法が定期的な支払いをサポートしていない
- 取引金額が最低閾値（例: BNPL方法）を下回っている

**解決策:** すべての適格要件が満たされていることを確認し、APIリクエストの設定を確認してください。
</Accordion>

<Accordion title="予期しない支払い方法が表示される">
**考えられる原因:**
- `allowed_payment_method_types`パラメータが正しく渡されていない
- マーチャントダッシュボードに競合する設定がある可能性があります

**解決策:** APIリクエストのペイロードを確認し、パラメータが文字列の配列として正しくフォーマットされていることを確認してください。
</Accordion>

<Accordion title="地域の支払い方法が利用できない">
**考えられる原因:**
- 顧客の所在地が支払い方法のサポート地域と一致しない
- 通貨が地域の支払い方法でサポートされていない
- マーチャントアカウントがその特定の支払い方法に対して有効になっていない可能性があります

**解決策:** 地域と通貨の互換性を確認し、その方法が利用可能であるべきだと考える場合はサポートに連絡してください。
</Accordion>

<Accordion title="インドの支払い方法のサブスクリプション問題">
**アップグレード/ダウングレード後にサブスクリプション料金が失敗した:**
- 新しい料金が既存のmandate制限（Rs 15,000の閾値）を超えている可能性があります
- 顧客は新しいmandateを確立するために支払い方法を更新する必要があるかもしれません

**解決策:** 顧客に支払い方法を更新するよう促すか、mandate制限内に収まるようにサブスクリプション金額を調整してください。

**料金の開始後にサブスクリプションが保留中:**
- 顧客が48時間の処理ウィンドウ中にmandateをキャンセルした可能性があります
- 顧客が高額の料金（≥Rs 15,000）を承認しなかった可能性があります

**解決策:** mandateのキャンセルや支払い失敗に関するWebhookイベントを監視します。リトライロジックを実装するか、顧客に支払い方法を再確立するよう促してください。

**支払いの引き落としが遅延している:**
- 通常の動作: インドのカードおよびUPIのサブスクリプションは、料金の開始と実際の引き落としの間に48時間の遅延があります
- この遅延は、銀行のAPIによってさらに2〜3時間延長される可能性があります

**解決策:** これは予期される動作です。支払い確認がWebhookを介して受信されるまで、特典やクレジットを有効にしないでください。
</Accordion>

<Accordion title="サブスクリプションの支払い方法の更新">
**支払い方法を更新するタイミング:**
- 顧客のカードが期限切れまたは交換された
- 顧客が別の支払い方法を使用したい
- サブスクリプションが失敗した支払いのために`on_hold`状態にある
- 何らかの理由で支払い方法を変更する必要がある

**アクティブなサブスクリプションの場合:**
アクティブなサブスクリプションの支払い方法を更新すると、将来の更新のために支払い方法が更新されます。即時の料金は発生しません。

**保留中のサブスクリプションの場合:**
サブスクリプションが`on_hold`状態（更新の失敗またはプラン変更料金の失敗）にある場合、支払い方法を更新すると：
- 残りの未払い金のために自動的に料金が作成されます
- その料金の請求書が生成されます
- 新しい支払い方法を使用して支払いが処理されます
- 支払いが成功した場合、サブスクリプションが`active`状態に再アクティブ化されます

<CodeGroup>

```javascript Node.js
// Update payment method for active subscription
await client.subscriptions.updatePaymentMethod('sub_123', {
  type: 'new',
  return_url: 'https://example.com/return'
});

// Or use existing payment method
await client.subscriptions.updatePaymentMethod('sub_123', {
  type: 'existing',
  payment_method_id: 'pm_abc123'
});

// For on_hold subscription - automatically creates charge
const response = await client.subscriptions.updatePaymentMethod('sub_123', {
  type: 'new',
  return_url: 'https://example.com/return'
});

if (response.payment_id) {
  // Charge created for remaining dues
  // Redirect customer to response.payment_link
  // Monitor webhooks: payment.succeeded → subscription.active
}
```

```python Python
# Update payment method for active subscription
client.subscriptions.update_payment_method(
    subscription_id="sub_123",
    type="new",
    return_url="https://example.com/return"
)

# Or use existing payment method
client.subscriptions.update_payment_method(
    subscription_id="sub_123",
    type="existing",
    payment_method_id="pm_abc123"
)

# For on_hold subscription - automatically creates charge
response = client.subscriptions.update_payment_method(
    subscription_id="sub_123",
    type="new",
    return_url="https://example.com/return"
)

if response.payment_id:
    # Charge created for remaining dues
    # Redirect customer to response.payment_link
    # Monitor webhooks: payment.succeeded → subscription.active
```

</CodeGroup>

**Webhookイベント:**
- `on_hold`サブスクリプションの場合: 支払い方法の更新が成功した後、`payment.succeeded`が受信され、その後`subscription.active`が続きます
- これらのイベントを監視して、サブスクリプションの再アクティブ化状況を追跡します
