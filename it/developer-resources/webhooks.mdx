I webhooks forniscono notifiche in tempo reale quando si verificano eventi specifici nel tuo account Dodo Payments. Utilizza i webhooks per automatizzare i flussi di lavoro, aggiornare il tuo database, inviare notifiche e mantenere i tuoi sistemi sincronizzati.

<Info>
La nostra implementazione dei webhook segue la specifica [Standard Webhooks](https://standardwebhooks.com/), garantendo compatibilità con le best practice del settore e con le librerie webhook esistenti.
</Info>

## Consegna del Webhook

### Timeout

I webhooks hanno una **finestra di timeout di 15 secondi** sia per le operazioni di connessione che di lettura. Assicurati che il tuo endpoint risponda rapidamente per evitare timeout.

<Tip>
Elabora i webhook in modo asincrono confermando immediatamente la ricezione con un codice di stato `200`, quindi gestisci l'elaborazione effettiva in background.
</Tip>

### Ritenti automatici

Se la consegna di un webhook fallisce, Dodo Payments riprova automaticamente con un backoff esponenziale per evitare di sovraccaricare il tuo sistema.

| Tentativo | Ritardo | Descrizione |
|---------|------------------------------|-------------|
| 1 | Immediatamente | Il primo tentativo avviene subito |
| 2 | 5 secondi | Secondo tentativo dopo un breve ritardo |
| 3 | 5 minuti | Terzo tentativo con backoff aumentato |
| 4 | 30 minuti | Quarto tentativo continuando il backoff |
| 5 | 2 ore | Quinto tentativo con ritardo esteso |
| 6 | 5 ore | Sesto tentativo con ritardo più lungo |
| 7 | 10 ore | Settimo tentativo con ritardo massimo |
| 8 | 10 ore | **Tentativo finale** - webhook contrassegnato come fallito se non riuscito |

<Info>
**Massimo di 8 tentativi di ritentativo** per ogni evento webhook. Ad esempio, se un webhook fallisce tre volte prima di riuscire, il tempo totale di consegna è di circa 35 minuti e 5 secondi dal primo tentativo.
</Info>

<Tip>
Usa la dashboard di Dodo Payments per ritentare manualmente singoli messaggi o recuperare in blocco tutti i messaggi falliti in qualsiasi momento.
</Tip>

### Idempotenza

Ogni evento webhook include un header unico `webhook-id`. Usa questo identificatore per implementare l'idempotenza ed evitare elaborazioni duplicate.

```javascript
// Example: Storing webhook IDs to prevent duplicate processing
const processedWebhooks = new Set();

app.post('/webhook', (req, res) => {
  const webhookId = req.headers['webhook-id'];
  
  if (processedWebhooks.has(webhookId)) {
    return res.status(200).json({ received: true });
  }
  
  processedWebhooks.add(webhookId);
  // Process the webhook...
});
```

<Warning>
Implementa sempre controlli di idempotenza. A causa dei ritentativi potresti ricevere lo stesso evento più volte.
</Warning>

### Ordinamento degli eventi

Gli eventi webhook possono arrivare in ordine errato a causa di ritentativi o condizioni di rete. Progetta il tuo sistema per gestire eventi in qualsiasi sequenza.

<Info>
Riceverai il **payload più recente al momento della consegna**, indipendentemente da quando l'evento webhook è stato emesso originariamente.
</Info>

## Sicurezza dei Webhook

Per garantire la sicurezza dei tuoi webhook, valida sempre i payload e utilizza HTTPS.

### Verifica delle firme

Ogni richiesta webhook include un header `webhook-signature`, una firma HMAC SHA256 del payload e del timestamp del webhook, firmata con la tua chiave segreta.

#### Verifica SDK (raccomandata)

Tutti gli SDK ufficiali includono helper integrati per validare e analizzare in modo sicuro i webhook in arrivo. Sono disponibili due metodi:

* `unwrap()`: Verifica le firme utilizzando la tua chiave segreta webhook
* `unsafe_unwrap()`: Analizza i payload senza verifica

<Tip>
  Fornisci la tua chiave segreta webhook tramite `DODO_PAYMENTS_WEBHOOK_KEY` quando inizializzi il client Dodo Payments.
</Tip>

<CodeGroup>

```typescript TypeScript/Node.js
import DodoPayments from 'dodopayments';
import express from 'express';

const app = express();
app.use(express.raw({ type: 'application/json' }));

const client = new DodoPayments({
  bearerToken: process.env.DODO_PAYMENTS_API_KEY,
  environment: process.env.DODO_PAYMENTS_ENVIRONMENT,
  webhookKey: process.env.DODO_PAYMENTS_WEBHOOK_KEY,
});

app.post('/webhook', async (req, res) => {
  try {
    const unwrapped = client.webhooks.unwrap(req.body.toString(), {
      headers: {
        'webhook-id': req.headers['webhook-id'] as string,
        'webhook-signature': req.headers['webhook-signature'] as string,
        'webhook-timestamp': req.headers['webhook-timestamp'] as string,
      },
    });
    res.json({ received: true });
  } catch (error) {
    res.status(401).json({ error: 'Invalid signature' });
  }
});
```

```python Python
from dodopayments import DodoPayments
import os

app = FastAPI()
client = DodoPayments(
```

```go Go
import (
	"context"
	"io"
	"net/http"
	"os"
	"github.com/dodopayments/dodopayments-go"
	"github.com/dodopayments/dodopayments-go/option"
)

func webhookHandler(w http.ResponseWriter, r *http.Request) {
	client := dodopayments.NewClient(
		option.WithBearerToken(os.Getenv("DODO_PAYMENTS_API_KEY")),
		option.WithEnvironment(os.Getenv("DODO_PAYMENTS_ENVIRONMENT")),
		option.WithWebhookKey(os.Getenv("DODO_PAYMENTS_WEBHOOK_KEY")),
	)
	rawBody, _ := io.ReadAll(r.Body)
	if _, err := client.Webhooks.Unwrap(context.Background(), rawBody, map[string]string{
		"webhook-id":       r.Header.Get("webhook-id"),
		"webhook-signature": r.Header.Get("webhook-signature"),
		"webhook-timestamp": r.Header.Get("webhook-timestamp"),
	}); err != nil {
		http.Error(w, "Invalid signature", http.StatusUnauthorized)
		return
	}
	w.WriteHeader(http.StatusOK)
}
```

#### Verifica manuale (alternativa)

Se non stai utilizzando un SDK, puoi verificare le firme tu stesso seguendo la specifica Standard Webhooks:

1. Costruisci il messaggio firmato concatenando `webhook-id`, `webhook-timestamp` e la stringa raw esatta JSON `payload`, separati da punti (`.`).
2. Calcola l'HMAC SHA256 di quella stringa usando la tua chiave segreta webhook della Dashboard.
3. Confronta la firma calcolata con l'header `webhook-signature`. Se coincidono, il webhook è autentico.

<Info>
  Seguiamo la specifica Standard Webhooks. Puoi usare le loro librerie per verificare le firme: [https://github.com/standard-webhooks/standard-webhooks/tree/main/libraries](https://github.com/standard-webhooks/standard-webhooks/tree/main/libraries). Per i formati dei payload degli eventi, consulta il [Webhook Payload](/developer-resources/webhooks/intents/payment).
</Info>

* Il tuo gestore webhook deve restituire un codice di stato `2xx status code` per confermare la ricezione dell'evento.
* Qualsiasi altra risposta sarà trattata come un errore e il webhook verrà ritentato.

## Struttura del Payload del Webhook

Comprendere la struttura del payload del webhook ti aiuta a analizzare e elaborare correttamente gli eventi.

### Formato della richiesta

```http
POST /your-webhook-url
Content-Type: application/json
```

### Intestazioni

<ParamField header="webhook-id" type="string" required>
Identificatore univoco per questo evento webhook. Usalo per i controlli di idempotenza.
</ParamField>

<ParamField header="webhook-signature" type="string" required>
Firma HMAC SHA256 per verificare l'autenticità del webhook.
</ParamField>

<ParamField header="webhook-timestamp" type="string" required>
Timestamp Unix (in secondi) quando il webhook è stato inviato.
</ParamField>

### Corpo della richiesta

<ResponseField name="business_id" type="string" required>
Identificativo della tua attività su Dodo Payments.
</ResponseField>

<ResponseField name="type" type="string" required>
Tipo di evento che ha attivato questo webhook (ad es. `payment.succeeded`, `subscription.created`).
</ResponseField>

<ResponseField name="timestamp" type="string" required>
Timestamp formattato ISO 8601 di quando l'evento è avvenuto.
</ResponseField>

<ResponseField name="data" type="object" required>
Payload specifico dell'evento contenente informazioni dettagliate sull'evento.

{/* LOCKED_PATTERN_0909fd3fa83cfc9cffddb2758977bdda */}
  <ResponseField name="payload_type" type="string">
  Tipo di risorsa: `Payment`, `Subscription`, `Refund`, `Dispute` o `LicenseKey`.
  </ResponseField>
  
  I campi aggiuntivi variano in base al tipo di evento. Consulta la documentazione specifica per eventi per gli schemi completi.
</Expandable>
</ResponseField>

```json
  "timestamp": "2024-01-01T12:00:00Z",

    "payload_type": "Payment | Subscription | Refund | Dispute | LicenseKey",
```

### Esempio di implementazione

Ecco un'implementazione completa di Express.js che mostra la verifica e la gestione dei webhook:

<CodeGroup>

```typescript Express.js Webhook Handler
import { Webhook } from "standardwebhooks";
import express from "express";

const app = express();
app.use(express.json());

const webhook = new Webhook(process.env.DODO_WEBHOOK_SECRET);

app.post('/webhook/dodo-payments', async (req, res) => {
  try {
    // Extract webhook headers
    const webhookHeaders = {
      "webhook-id": req.headers["webhook-id"] as string,
      "webhook-signature": req.headers["webhook-signature"] as string,
      "webhook-timestamp": req.headers["webhook-timestamp"] as string,
    };

    // Verify the webhook signature
    const payload = JSON.stringify(req.body);
    await webhook.verify(payload, webhookHeaders);
    
    // Acknowledge receipt immediately
    res.status(200).json({ received: true });
    
    // Process webhook asynchronously
    processWebhookAsync(req.body).catch(console.error);
    
  } catch (error) {
    console.error('Webhook verification failed:', error);
    res.status(400).json({ error: 'Invalid signature' });
  }
});

async function processWebhookAsync(data: any) {
  // Handle the webhook event based on type
  switch (data.type) {
    case 'payment.succeeded':
      await handlePaymentSucceeded(data);
      break;
    case 'subscription.created':
      await handleSubscriptionCreated(data);
      break;
    // Add more event handlers...
  }
}
```

## Impostazioni avanzate

La scheda Impostazioni avanzate fornisce opzioni di configurazione aggiuntive per perfezionare il comportamento del tuo endpoint webhook.

### Limitazione della velocità (Throttling)

Controlla la velocità con cui gli eventi webhook vengono consegnati al tuo endpoint per evitare di sovraccaricare il sistema.

<Steps>
{/* LOCKED_PATTERN_0d60e6c1f9ea51c3bad513723bbb4ee3 */}
    Nella scheda **Avanzate**, individua la sezione "Rate Limit (throttling)".
</Step>

{/* LOCKED_PATTERN_0b32e00ee8468aab284cb8d5e277e1f7 */}
    Clicca sul pulsante **Modifica** per modificare le impostazioni del rate limit.
    
    <Info>
    Di default, i webhook hanno applicato "Nessun limite di velocità", quindi gli eventi vengono consegnati non appena si verificano.
    </Info>
</Step>

{/* LOCKED_PATTERN_5e44ce72dcc720e1fd8525ede5e25905 */}
    Configura il limite di velocità desiderato per controllare la frequenza di consegna dei webhook e prevenire il sovraccarico del sistema.
</Step>
</Steps>

<Tip>
Usa la limitazione della velocità quando il tuo gestore webhook ha bisogno di tempo per elaborare gli eventi o quando vuoi raggruppare più eventi insieme.
</Tip>

### Header personalizzati

Aggiungi header HTTP personalizzati a tutte le richieste webhook inviate al tuo endpoint. Questo è utile per autenticazione, instradamento o aggiunta di metadati alle richieste webhook.

<Steps>
{/* LOCKED_PATTERN_c841f58a63f13da8452dd9cb10f43cff */}
    Nella sezione "Header personalizzati", inserisci una **Chiave** e un **Valore** per il tuo header personalizzato.
</Step>

{/* LOCKED_PATTERN_704f232c5d005afeff9cb50ea2d5abe2 */}
    Clicca sul pulsante **+** per aggiungere ulteriori header personalizzati secondo necessità.
</Step>

{/* LOCKED_PATTERN_08395c29a4a5c780a5cf3de53acf78b9 */}
    I tuoi header personalizzati verranno inclusi in tutte le richieste webhook verso questo endpoint.
</Step>
</Steps>

### Trasformazioni

Le trasformazioni ti permettono di modificare il payload di un webhook e reindirizzarlo verso un URL diverso. Questa funzionalità potente ti consente di:

- Modificare la struttura del payload prima dell'elaborazione
- Instradare i webhook verso endpoint diversi in base al contenuto
- Aggiungere o rimuovere campi dal payload
- Trasformare i formati dei dati

<Steps>
{/* LOCKED_PATTERN_b50ea3f5e28aae978fe2be49c50f40bc */}
    Attiva l'interruttore **Enabled** per abilitare la funzionalità di trasformazione.
</Step>

{/* LOCKED_PATTERN_4ac4616f59ca1d74262754150410c05f */}
    Clicca su **Edit transformation** per definire le regole di trasformazione.
    
    <Info>
    Puoi usare JavaScript per trasformare il payload del webhook e specificare un URL di destinazione diverso.
    </Info>
</Step>

{/* LOCKED_PATTERN_4541cb6ebe3833f9d443467db6f92a42 */}
    Usa l'interfaccia di test per verificare che la trasformazione funzioni correttamente prima di metterla in produzione.
</Step>
</Steps>

<Warning>
Le trasformazioni possono influire significativamente sulle prestazioni di consegna dei webhook. Testale a fondo e mantieni la logica semplice ed efficiente.
</Warning>

<Tip>
Le trasformazioni sono particolarmente utili per:
- Convertire tra formati di dati diversi
- Filtrare gli eventi in base a criteri specifici
- Aggiungere campi calcolati al payload
- Instradare eventi verso microservizi differenti
</Tip>
