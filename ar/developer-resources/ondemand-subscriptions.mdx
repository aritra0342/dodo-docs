## نظرة عامة

تتيح لك الاشتراكات عند الطلب تفويض وسيلة دفع العميل مرة واحدة ثم تحصيل مبالغ متغيرة كلما احتجت، بدلاً من جدول زمني ثابت.

<Info>
قد تحتاج هذه الميزة إلى تفعيلها على حسابك. اتصل بالدعم إذا لم ترها في لوحة التحكم الخاصة بك.
</Info>

استخدم هذا الدليل لـ:
- إنشاء اشتراك عند الطلب (تفويض تفويض مع سعر أولي اختياري)
- تفعيل رسوم لاحقة بمبالغ مخصصة
- تتبع النتائج باستخدام الويب هوكس

لإعداد اشتراك عام، راجع [دليل تكامل الاشتراك](/developer-resources/subscription-integration-guide).

## المتطلبات الأساسية

- حساب تاجر Dodo Payments ومفتاح API
- سر الويب هوك مُعد ونقطة نهاية لاستقبال الأحداث
- منتج اشتراك في الكتالوج الخاص بك

<Tip>
إذا كنت تريد من العميل الموافقة على التفويض عبر الخروج المستضاف، قم بتعيين `payment_link: true` وقدم `return_url`.
</Tip>

## كيف يعمل الاشتراك عند الطلب

1. تقوم بإنشاء اشتراك باستخدام كائن `on_demand` لتفويض وسيلة الدفع وجمع رسوم أولية اختيارية.
2. لاحقًا، تقوم بإنشاء رسوم ضد ذلك الاشتراك بمبالغ مخصصة باستخدام نقطة نهاية الرسوم المخصصة.
3. تستمع إلى الويب هوكس (مثل `payment.succeeded`، `payment.failed`) لتحديث نظامك.

## إنشاء اشتراك عند الطلب

نقطة النهاية: [POST /checkouts](/api-reference/checkout-sessions/create)

حقول الطلب الرئيسية (الجسم):  
يرجى العثور عليها في [إنشاء جلسة الخروج](/api-reference/checkout-sessions/create)

### إنشاء اشتراك عند الطلب

<Tab title="Node.js SDK">

```javascript
import DodoPayments from 'dodopayments';

const client = new DodoPayments({
  bearerToken: process.env.DODO_PAYMENTS_API_KEY,
  environment: 'test_mode', // defaults to 'live_mode'
});

async function main() {
  const subscription = await client.checkoutSessions.create({
    product_cart: [{ product_id: 'pdt_123', quantity: 1 }],
    billing_address:  { city: 'SF', country: 'US', state: 'CA', street: '1 Market St', zipcode: '94105' },
    customer: { customer_id: 'cus_123' },
    return_url: 'https://example.com/billing/success',
    subscription_data: {
        on_demand: {
            mandate_only: true // set false to collect an initial charge
            // product_price: 1000, // optional: charge $10.00 now if mandate_only is false
            // product_currency: 'USD',
            // product_description: 'Custom initial charge',
            // adaptive_currency_fees_inclusive: false,
        }
    }
  });

  console.log(subscription.checkout_url);
}

main().catch(console.error);
```

import (

## تحصيل اشتراك عند الطلب

بعد تفويض التفويض، قم بإنشاء رسوم حسب الحاجة.

نقطة النهاية: [POST /subscriptions/\{subscription_id\}/charge](/api-reference/subscriptions/create-charge)

حقول الطلب الرئيسية (الجسم):

<AccordionGroup>
  <Accordion title="معلمات جسم طلب الرسوم">
    <ParamField body="product_price" type="integer" required>
      المبلغ المراد تحصيله (بالوحدة الأصغر للعملة). مثال: لتحصيل 25.00 دولار، مرر <code>2500</code>.
    </ParamField>

    <ParamField body="product_currency" type="string">
      تجاوز العملة الاختياري للرسوم.
    </ParamField>

    <ParamField body="product_description" type="string">
      تجاوز الوصف الاختياري لهذه الرسوم.
    </ParamField>

    <ParamField body="adaptive_currency_fees_inclusive" type="boolean">
      إذا كانت صحيحة، تشمل رسوم العملة التكيفية ضمن <code>product_price</code>. إذا كانت خاطئة، تتم إضافة الرسوم فوق ذلك.
    </ParamField>

    <ParamField body="metadata" type="object">
      بيانات وصفية إضافية للدفع. إذا تم حذفها، يتم استخدام بيانات الاشتراك الوصفية.
    </ParamField>
  </Accordion>
</AccordionGroup>

## محاولات الدفع

قد يقوم نظام كشف الاحتيال لدينا بحظر أنماط المحاولة العدوانية (ويمكن أن يميزها كاختبار محتمل للبطاقة). اتبع سياسة إعادة المحاولة الآمنة.

<Warning>
يمكن أن يتم تمييز أنماط المحاولة المتفجرة كاحتيالية أو كاختبار محتمل للبطاقة من قبل أنظمة المخاطر ومعالجاتنا. تجنب المحاولات المتجمعة؛ اتبع جدول التراجع وإرشادات محاذاة الوقت أدناه.
</Warning>

### مبادئ سياسة إعادة المحاولة الآمنة

- **آلية التراجع**: استخدم التراجع الأسي بين المحاولات.
- **حدود إعادة المحاولة**: حدد إجمالي المحاولات (3-4 محاولات كحد أقصى).
- **تصفية ذكية**: أعد المحاولة فقط في حالات الفشل القابلة لإعادة المحاولة (مثل أخطاء الشبكة/المصدر، نقص الأموال)؛ لا تعيد المحاولة أبدًا في حالات الرفض الصعبة.
- **منع اختبار البطاقة**: لا تعيد المحاولة في حالات الفشل مثل `DO_NOT_HONOR`، `STOLEN_CARD`، `LOST_CARD`، `PICKUP_CARD`، `FRAUDULENT`، `AUTHENTICATION_FAILURE`.
- **تغيير البيانات الوصفية (اختياري)**: إذا كنت تحتفظ بنظام إعادة المحاولة الخاص بك، قم بتمييز المحاولات عبر البيانات الوصفية (مثل `retry_attempt`).

### جدول إعادة المحاولة المقترح (الاشتراكات)

- **المحاولة الأولى**: فورية عند إنشاء الرسوم
- **المحاولة الثانية**: بعد 3 أيام
- **المحاولة الثالثة**: بعد 7 أيام أخرى (10 أيام إجمالاً)
- **المحاولة الرابعة (الأخيرة)**: بعد 7 أيام أخرى (17 يومًا إجمالاً)

الخطوة النهائية: إذا لم يتم الدفع بعد، قم بوضع علامة على الاشتراك كغير مدفوع أو قم بإلغائه، بناءً على سياستك. قم بإخطار العميل خلال الفترة الزمنية لتحديث وسيلة الدفع الخاصة بهم.

### تجنب المحاولات المتفجرة؛ التوافق مع وقت التفويض

- قم بتثبيت المحاولات على الطابع الزمني الأصلي للتفويض لتجنب سلوك "الانفجار" عبر محفظتك.
- مثال: إذا بدأ العميل تجربة أو تفويض في الساعة 1:10 مساءً اليوم، قم بجدولة المحاولات المتابعة في الساعة 1:10 مساءً في الأيام التالية وفقًا لجدول التراجع الخاص بك (مثل +3 أيام → 1:10 مساءً، +7 أيام → 1:10 مساءً).
- بدلاً من ذلك، إذا كنت تخزن وقت الدفع الناجح الأخير `T`، قم بجدولة المحاولة التالية في `T + X days` للحفاظ على محاذاة وقت اليوم.

<Note>
المنطقة الزمنية وDST: استخدم معيار وقت ثابت للتخطيط وقم بالتحويل للعرض فقط للحفاظ على الفترات.
</Note>

### رموز الرفض التي يجب ألا تعيد المحاولة عليها

- `STOLEN_CARD`
- `DO_NOT_HONOR`
- `FRAUDULENT`
- `PICKUP_CARD`
- `AUTHENTICATION_FAILURE`
- `LOST_CARD`

<Note>
للحصول على قائمة شاملة بأسباب الرفض وما إذا كانت قابلة للتصحيح من قبل المستخدم، راجع
[فشل المعاملات](/api-reference/transaction-failures) الوثائق.
</Note>

<Tip>
قم بإعادة المحاولة فقط في حالات المشاكل المؤقتة/الناعمة (مثل `insufficient_funds`، `issuer_unavailable`، `processing_error`، مهلات الشبكة). إذا تكررت نفس حالة الرفض، توقف عن المزيد من المحاولات.
</Tip>

### إرشادات التنفيذ (بدون كود)

- استخدم جدولة/طابور يحتفظ بطوابع زمنية دقيقة؛ احسب المحاولة التالية في نفس وقت اليوم (مثل `T + 3 days` في نفس HH:MM).
- احتفظ وارجع إلى طابع الزمن الخاص بآخر دفع ناجح `T` لحساب المحاولة التالية؛ لا تجمع بين اشتراكات متعددة في نفس اللحظة.
- قم دائمًا بتقييم سبب الرفض الأخير؛ توقف عن المحاولات في حالات الرفض الصعبة في قائمة التجاوز أعلاه.
- حدد المحاولات المتزامنة لكل عميل وحساب لمنع الارتفاعات العرضية.
- تواصل بشكل استباقي: أرسل بريدًا إلكترونيًا/SMS للعميل لتحديث وسيلة الدفع الخاصة بهم قبل المحاولة المجدولة التالية.
- استخدم البيانات الوصفية فقط للرؤية (مثل `retry_attempt`)؛ لا تحاول "التملص" من أنظمة الاحتيال/المخاطر عن طريق تدوير الحقول غير المهمة.

## تتبع النتائج باستخدام الويب هوكس

قم بتنفيذ معالجة الويب هوكس لتتبع رحلة العميل. راجع [تنفيذ الويب هوكس](/developer-resources/integration-guide#implementing-webhooks).

- **subscription.active**: تم تفويض التفويض وتم تفعيل الاشتراك
- **subscription.failed**: فشل الإنشاء (مثل فشل التفويض)
- **subscription.on_hold**: تم وضع الاشتراك في حالة الانتظار (مثل الحالة غير المدفوعة)
- **payment.succeeded**: تم تحصيل الرسوم بنجاح
- **payment.failed**: فشل تحصيل الرسوم

<Tip>
بالنسبة للتدفقات عند الطلب، ركز على `payment.succeeded` و`payment.failed` لمصالحة الرسوم المستندة إلى الاستخدام.
</Tip>

## الاختبار والخطوات التالية

<Steps>
<Step title="إنشاء في وضع الاختبار">
استخدم مفتاح API الخاص بك للاختبار لإنشاء الاشتراك باستخدام `payment_link: true`، ثم افتح الرابط وأكمل التفويض.
</Step>
<Step title="تحفيز رسوم">
استدعِ نقطة نهاية الرسوم بمبلغ صغير `product_price` (مثل `100`) وتحقق من أنك تتلقى `payment.succeeded`.
</Step>
<Step title="الانتقال إلى الوضع المباشر">
قم بالتبديل إلى مفتاح API المباشر الخاص بك بمجرد أن تكون قد تحقق من الأحداث وتحديثات الحالة الداخلية.
</Step>
</Steps>

## استكشاف الأخطاء وإصلاحها

- **422 طلب غير صالح**: تأكد من أن `on_demand.mandate_only` تم توفيره عند الإنشاء و`product_price` تم توفيره للرسوم.
- **أخطاء العملة**: إذا قمت بتجاوز `product_currency`، تأكد من أنه مدعوم لحسابك وعميلك.
- **لم يتم استلام الويب هوكس**: تحقق من عنوان URL الخاص بك وتكوين سر التوقيع.
