## Ikhtisar

Langganan sesuai permintaan memungkinkan Anda untuk mengotorisasi metode pembayaran pelanggan sekali dan kemudian mengenakan jumlah variabel kapan pun Anda perlu, alih-alih pada jadwal tetap. Fitur ini tersedia untuk semua akunâ€”tidak memerlukan persetujuan.

Gunakan panduan ini untuk:
- Membuat langganan sesuai permintaan (mengotorisasi mandat dengan harga awal opsional)
- Memicu biaya berikutnya dengan jumlah kustom
- Melacak hasil menggunakan webhook

Untuk pengaturan langganan umum, lihat [Panduan Integrasi Langganan](/developer-resources/subscription-integration-guide).

## Prasyarat

- Akun pedagang Dodo Payments dan kunci API
- Rahasia webhook yang dikonfigurasi dan endpoint untuk menerima peristiwa
- Produk langganan dalam katalog Anda

<Tip>\nJika Anda ingin pelanggan menyetujui mandat melalui hosted checkout, atur `payment_link: true` dan sediakan `return_url`.\n</Tip>

## Cara kerja sesuai permintaan

1. Anda membuat langganan dengan objek `on_demand` untuk mengotorisasi metode pembayaran dan secara opsional mengumpulkan biaya awal.\n2. Nanti, Anda membuat tagihan terhadap langganan tersebut dengan jumlah yang disesuaikan menggunakan endpoint tagihan khusus.\n3. Anda mendengarkan webhook (misalnya `payment.succeeded`, `payment.failed`) untuk memperbarui sistem Anda.

```javascript
import DodoPayments from 'dodopayments';

const client = new DodoPayments({
  bearerToken: process.env.DODO_PAYMENTS_API_KEY,
  environment: 'test_mode', // defaults to 'live_mode'
});

async function main() {
  const subscription = await client.checkoutSessions.create({
    product_cart: [{ product_id: 'pdt_123', quantity: 1 }],
    billing_address:  { city: 'SF', country: 'US', state: 'CA', street: '1 Market St', zipcode: '94105' },
    customer: { customer_id: 'cus_123' },
    return_url: 'https://example.com/billing/success',
    subscription_data: {
        on_demand: {
            mandate_only: true // set false to collect an initial charge
            // product_price: 1000, // optional: charge $10.00 now if mandate_only is false
            // product_currency: 'USD',
            // product_description: 'Custom initial charge',
            // adaptive_currency_fees_inclusive: false,
        }
    }
  });

  console.log(subscription.checkout_url);
}

main().catch(console.error);
```

## Pengulangan pembayaran

Sistem deteksi penipuan kami mungkin memblokir pola pengulangan yang agresif (dan dapat menandainya sebagai pengujian kartu potensial). Ikuti kebijakan pengulangan yang aman.

<Warning>\nPolanya percobaan ulang beruntun dapat dianggap sebagai penipuan atau dugaan pengujian kartu oleh sistem risiko dan prosesor kami. Hindari percobaan ulang yang terkonsentrasi; ikuti jadwal backoff dan panduan penyelarasan waktu di bawah.\n</Warning>

### Prinsip untuk kebijakan pengulangan yang aman

- **Mekanisme backoff**: Gunakan backoff eksponensial antara percobaan ulang.\n- **Batas percobaan ulang**: Batasi total percobaan ulang (maksimal 3â€“4 kali percobaan).\n- **Penyaringan cerdas**: Lakukan percobaan ulang hanya pada kegagalan yang bisa dicoba ulang (misalnya, kesalahan jaringan/penerbit, dana tidak mencukupi); jangan pernah mengulang penolakan keras.\n- **Pencegahan pengujian kartu**: Jangan ulang kegagalan seperti `DO_NOT_HONOR`, `STOLEN_CARD`, `LOST_CARD`, `PICKUP_CARD`, `FRAUDULENT`, `AUTHENTICATION_FAILURE`.\n- **Variasikan metadata (opsional)**: Jika Anda memelihara sistem retry sendiri, bedakan percobaan ulang lewat metadata (misalnya `retry_attempt`).

### Jadwal pengulangan yang disarankan (langganan)

- **Percobaan ke-1**: Segera setelah Anda membuat biaya
- **Percobaan ke-2**: Setelah 3 hari
- **Percobaan ke-3**: Setelah 7 hari lagi (total 10 hari)
- **Percobaan ke-4 (akhir)**: Setelah 7 hari lagi (total 17 hari)

Langkah terakhir: jika masih belum dibayar, tandai langganan sebagai tidak dibayar atau batalkan, berdasarkan kebijakan Anda. Beri tahu pelanggan selama jendela untuk memperbarui metode pembayaran mereka.

### Hindari pengulangan mendadak; sesuaikan dengan waktu otorisasi

- Sematkan percobaan ulang pada cap waktu otorisasi asli untuk menghindari perilaku "burst" di seluruh portofolio Anda.\n- Contoh: Jika pelanggan memulai uji coba atau mandat pada pukul 1:10 siang hari ini, jadwalkan percobaan ulang lanjutan pada pukul 1:10 siang di hari-hari berikutnya sesuai backoff Anda (misalnya, +3 hari â†’ 1:10 siang, +7 hari â†’ 1:10 siang).\n- Atau, jika Anda menyimpan waktu pembayaran terakhir yang berhasil `T`, jadwalkan percobaan berikutnya pada `T + X days` untuk mempertahankan keselarasan waktu dalam hari.

<Note>\nZona waktu dan DST: gunakan standar waktu yang konsisten untuk penjadwalan dan ubah hanya untuk tampilan agar interval tetap terjaga.\n</Note>

### Kode penolakan yang sebaiknya tidak Anda ulangi

- `STOLEN_CARD`\n- `DO_NOT_HONOR`\n- `FRAUDULENT`\n- `PICKUP_CARD`\n- `AUTHENTICATION_FAILURE`\n- `LOST_CARD`

<Note>\nUntuk daftar lengkap alasan penolakan dan apakah alasan tersebut dapat diperbaiki pengguna, lihat dokumentasi\n[Transaction Failures](/api-reference/transaction-failures).\n</Note>

<Tip>\nLakukan percobaan ulang hanya pada masalah lunak/sementara (misalnya `insufficient_funds`, `issuer_unavailable`, `processing_error`, batas waktu jaringan). Jika penolakan yang sama terulang, jeda percobaan ulang selanjutnya.\n</Tip>

### Pedoman implementasi (tanpa kode)

- Gunakan penjadwal/antrian yang menyimpan cap waktu secara tepat; hitung percobaan berikutnya pada offset waktu-hari yang sama (misalnya `T + 3 days` pada HH:MM yang sama).\n- Simpan dan rujuk cap waktu pembayaran terakhir yang berhasil `T` untuk menghitung percobaan berikutnya; jangan satukan banyak langganan pada saat yang sama.\n- Selalu evaluasi alasan penolakan terakhir; hentikan percobaan ulang untuk penolakan keras dalam daftar yang diabaikan di atas.\n- Batasi percobaan ulang bersamaan per pelanggan dan per akun untuk mencegah lonjakan tidak sengaja.\n- Komunikasikan secara proaktif: kirim email/SMS kepada pelanggan agar memperbarui metode pembayaran mereka sebelum percobaan berikutnya dijadwalkan.\n- Gunakan metadata hanya untuk observabilitas (misalnya `retry_attempt`); jangan pernah mencoba "mengelabui" sistem penipuan/risiko dengan mengganti bidang yang tidak penting.

## Lacak hasil dengan webhook

Implementasikan penanganan webhook untuk melacak perjalanan pelanggan. Lihat [Implementasi Webhook](/developer-resources/integration-guide#implementing-webhooks).

- **subscription.active**: Mandat diotorisasi dan langganan diaktifkan
- **subscription.failed**: Pembuatan gagal (misalnya, kegagalan mandat)
- **subscription.on_hold**: Langganan ditangguhkan (misalnya, status tidak dibayar)
- **payment.succeeded**: Biaya berhasil
- **payment.failed**: Biaya gagal

<Tip>\nUntuk alur on-demand, fokus pada `payment.succeeded` dan `payment.failed` untuk merekonsiliasi tagihan berbasis penggunaan.\n</Tip>

## Pemecahan masalah

- **422 Invalid Request**: Pastikan `on_demand.mandate_only` disediakan saat pembuatan dan `product_price` disediakan untuk tagihan.\n- **Kesalahan mata uang**: Jika Anda menimpa `product_currency`, pastikan itu didukung untuk akun dan pelanggan Anda.\n- **Tidak menerima webhook**: Verifikasi konfigurasi URL webhook dan rahasia tanda tangannya.
