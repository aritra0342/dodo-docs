## Comment cela fonctionne-t-il ?

Les méthodes de paiement sont automatiquement présentées aux clients en fonction de :

- **Localisation du client** — Les méthodes régionales comme iDEAL (Pays-Bas) ou UPI (Inde) n'apparaissent que pour les clients dans ces régions
- **Monnaie de transaction** — Les méthodes de paiement sont filtrées par les monnaies supportées
- **Type de dispositif** — Apple Pay n'apparaît que sur les dispositifs et navigateurs Apple compatibles
- **Type de transaction** — Certaines méthodes ne prennent pas en charge les abonnements ou ont des montants minimums

<Info>
**Activation automatique :** Toutes les méthodes de paiement sont pré-configurées et gérées par Dodo Payments. Pas de configuration supplémentaire nécessaire — elles deviennent disponibles une fois que votre compte est vérifié et que le mode en direct est activé.
</Info>

## Configurer les méthodes de paiement

Par défaut, toutes les méthodes de paiement éligibles apparaissent automatiquement. Utilisez `allowed_payment_method_types` pour restreindre quelles méthodes sont affichées :

<Tabs>
<Tab title="JavaScript">

```javascript
const session = await client.checkoutSessions.create({
  product_cart: [{ product_id: 'prod_123', quantity: 1 }],
  allowed_payment_method_types: [
    'credit',
    'debit', 
    'apple_pay',
    'google_pay'
  ],
  return_url: 'https://example.com/success'
});
```

</Tab>

<Tab title="Python">

```python
session = client.checkout_sessions.create(
    product_cart=[{"product_id": "prod_123", "quantity": 1}],
    allowed_payment_method_types=[
        "credit",
        "debit",
        "apple_pay", 
        "google_pay"
    ],
    return_url="https://example.com/success"
)
```

</Tab>

<Tab title="cURL">

```bash
curl -X POST https://test.dodopayments.com/checkout-sessions \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "product_cart": [{"product_id": "prod_123", "quantity": 1}],
    "allowed_payment_method_types": ["credit", "debit", "apple_pay", "google_pay"],
    "return_url": "https://example.com/success"
  }'
```

</Tab>
</Tabs>

### Types de méthodes disponibles

| Type | Description |
| :--- | :---------- |
| `credit` | Cartes de crédit (Visa, Mastercard, Amex, etc.) |
| `debit` | Cartes de débit |
| `apple_pay` | Apple Pay |
| `google_pay` | Google Pay |
| `amazon_pay` | Amazon Pay |
| `cashapp` | Cash App Pay |
| `klarna` | Klarna BNPL |
| `afterpay_clearpay` | Afterpay / Clearpay |
| `upi_collect` | UPI Collect (QR code) |
| `ideal` | iDEAL (Pays-Bas) |
| `bancontact_card` | Bancontact (Belgique) |
| `eps` | EPS (Autriche) |
| `multibanco` | Multibanco (Portugal) |
| `revolut_pay` | RevolutPay |
| `billie` | Billie B2B BNPL |

<Warning>
Incluez toujours `credit` et `debit` comme solutions de secours. Si toutes les méthodes de paiement spécifiées ne sont pas disponibles pour un client, le processus de paiement échouera.
</Warning>

## Bonnes pratiques

<AccordionGroup>
<Accordion title="Adapter les méthodes à votre marché">
Incluez des méthodes de paiement régionales pour vos marchés cibles. iDEAL capture ~60 % des paiements en ligne aux Pays-Bas ; UPI domine en Inde. Omettre ces méthodes nuit considérablement à la conversion.
</Accordion>

<Accordion title="Toujours fournir des solutions de secours par carte">
Même en ciblant des méthodes spécifiques, incluez `credit` et `debit` comme solutions de secours. Tous les clients n'ont pas accès à chaque méthode de paiement.
</Accordion>

<Accordion title="Considérer les minimums de transaction">
Les méthodes BNPL (Klarna, Afterpay) nécessitent un minimum de 50,01 $. Ne les incluez pas pour les produits de faible valeur — elles n'apparaîtront tout simplement pas, ce qui est acceptable.
</Accordion>

<Accordion title="Tester dans différentes régions et sur différents dispositifs">
Utilisez le mode test pour vérifier la disponibilité des méthodes de paiement pour différents emplacements de clients, monnaies et types de dispositifs avant de passer en direct.
</Accordion>
</AccordionGroup>

## Dépannage

<AccordionGroup>
<Accordion title="Méthode de paiement non apparente">
**Vérifiez ces facteurs :**
1. La méthode est-elle incluse dans `allowed_payment_method_types` ?
2. La localisation du client correspond-elle à la région prise en charge par la méthode ?
3. La monnaie de transaction est-elle supportée par la méthode ?
4. Le montant de la transaction respecte-t-il le minimum (par exemple, 50,01 $ pour BNPL) ?
5. Pour les abonnements : la méthode prend-elle en charge les paiements récurrents ?

**Solution :** Retirez temporairement `allowed_payment_method_types` pour voir toutes les méthodes éligibles, puis rétrécissez le problème.
</Accordion>

<Accordion title="Méthodes inattendues apparaissant">
**Causes possibles :**
- `allowed_payment_method_types` n'est pas passé correctement
- Le paramètre est mal formaté (doit être un tableau de chaînes)

**Solution :** Consignez votre charge utile de requête API et vérifiez la structure du paramètre.
</Accordion>

<Accordion title="Méthode régionale non disponible">
**Vérifiez :**
- L'adresse de facturation du client correspond-elle à la région de la méthode
- La monnaie est-elle correctement réglée (par exemple, EUR pour iDEAL)
- La devise adaptive est-elle activée si le client est international

**Solution :** Vérifiez le code de pays de l'adresse de facturation et la monnaie dans votre requête API.
</Accordion>

<Accordion title="Problèmes de méthode de paiement indienne">
Les abonnements UPI et Rupay ont des exigences uniques, y compris des délais de traitement de 48 heures et des limites de mandats de la RBI.

**Voir :** <a href="/features/payment-methods/india">Méthodes de paiement en Inde</a> pour des conseils complets.
</Accordion>
</AccordionGroup>
