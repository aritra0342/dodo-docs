Dodo Payments 支持多种支付方式，以帮助您为全球客户提供服务。所有列出的支付方式在您的商户账户验证通过并启用实时模式后会自动可用。

<Info>
**自动激活：** 您无需进行额外配置。所有支付方式均由 Dodo Payments 预配置和管理。您可以选择使用下面描述的 API 配置来限制在结账时显示哪些方式。
</Info>

## 配置支付方式

虽然所有支付方式会根据客户的位置和货币自动提供，但您可以选择使用 `allowed_payment_method_types` 参数限制结账时显示的支付选项。

### 何时配置支付方式

您可能希望配置允许的支付方式以：

虽然所有支付方式会根据客户的位置和货币自动可用，但您可以选择性地使用 `allowed_payment_method_types` 参数限制结账时显示的支付选项。

### 实施

<Steps>
<Step title="将参数添加到 API 请求">
在您的支付或订阅创建请求中包含 `allowed_payment_method_types` 字段：

```json
{
  "allowed_payment_method_types": [
    "credit",
    "debit",
    "apple_pay"
  ]
}
```

</Step>

<Steps>
<Step title="向API请求添加参数">
在您的支付或订阅创建请求中包含 `allowed_payment_method_types` 字段：

```json
{
  "allowed_payment_method_types": [
    "credit",
    "debit",
    "apple_pay"
  ]
}
```

<Accordion title="区域考虑因素">
在选择支付方式时考虑您的目标市场。包括特定区域的选项（如印度的 UPI 或荷兰的 iDEAL）可以显著提高这些市场的转化率。
</Accordion>

<Accordion title="监控和优化">
跟踪客户最常使用的支付方式，并根据实际使用模式和转化数据调整允许的方式。
</Accordion>
</AccordionGroup>

<AccordionGroup>
<Accordion title="默认行为">
如果未指定 `allowed_payment_method_types`，所有符合条件的支付方式将根据客户的位置、货币和交易类型可用。仅在需要限制支付方式时使用此参数。
</Accordion>

支付方式的可用性取决于多个因素：
- **客户位置**：某些支付方式是区域特定的
- **商户设置**：您的账户配置和能力
- **货币**：支持的支付方式因货币而异
- **交易类型**：某些方法不支持订阅或有最低金额要求

<Note>
将支付方式添加到 `allowed_payment_method_types` 并不保证它会出现 - 它还必须根据这些因素符合资格。
</Note>

## 故障排除

<AccordionGroup>
<Accordion title="结账时未出现支付方式">
**可能原因：**
- 支付方式未包含在 `allowed_payment_method_types` 中
- 支付方式不支持客户的位置
- 支付方式不支持交易货币
- 对于订阅：支付方式不支持循环支付
- 交易金额低于最低阈值（例如，BNPL 方法）

<Note>
将支付方式添加到 `allowed_payment_method_types` 并不能保证它会出现 - 它还必须根据这些因素符合条件。
</Note>

<Accordion title="意外出现的支付方式">
**可能原因：**
- `allowed_payment_method_types` 参数未正确传递
- 您的商户仪表板中可能存在冲突的配置

<AccordionGroup>
<Accordion title="结账时支付方式未出现">
**可能原因：**
- 支付方式未包含在 `allowed_payment_method_types`
- 支付方式不支持客户的位置
- 支付方式不支持交易货币
- 对于订阅：支付方式不支持定期付款
- 交易金额低于最低阈值（例如，BNPL 方法）

<Accordion title="区域支付方式不可用">
**可能原因：**
- 客户的位置与支付方式支持的区域不匹配
- 货币不被区域支付方式支持
- 您的商户账户可能未启用该特定支付方式

<Accordion title="意外支付方式出现">
**可能原因：**
- `allowed_payment_method_types` 参数未正确传递
- 您的商户仪表板中可能存在冲突的配置

<Accordion title="印度支付方式订阅问题">
**升级/降级后订阅收费失败：**
- 新的收费金额可能超过现有授权限制（15000 卢比阈值）
- 客户可能需要更新其支付方式以建立新的授权

**解决方案：** 提示客户更新其支付方式或调整订阅金额以保持在授权限制内。

**收费启动后订阅暂停：**
- 客户可能在 48 小时处理窗口内取消了授权
- 客户可能未能授权高价值收费（≥15000 卢比）

**解决方案：** 监控授权取消和支付失败的 webhook 事件。实施重试逻辑或提示客户重新建立其支付方式。

**支付扣款延迟：**
- 正常行为：印度卡和 UPI 订阅在收费启动和实际扣款之间有 48 小时的延迟
- 根据银行 API，此延迟可能延长至 2-3 小时

**解决方案：** 这是预期的行为。在通过 webhook 收到支付确认之前，请勿激活福利或积分。
</Accordion>

<Accordion title="更新订阅的支付方式">
**何时更新支付方式：**
- 客户的卡过期或被替换
- 客户希望使用不同的支付方式
- 由于支付失败，订阅处于 `on_hold` 状态
- 由于任何原因需要更改支付方式

**对于活跃的订阅：**
更新活跃订阅的支付方式仅会更新未来续订的支付方式。不会立即产生收费。

**对于暂停的订阅：**
当订阅处于 `on_hold` 状态（由于续订失败或计划变更收费失败），更新支付方式将：
- 自动为剩余欠款创建收费
- 为收费生成发票
- 使用新支付方式处理支付
- 在成功支付后将订阅重新激活为 `active` 状态

<Accordion title="更新订阅的支付方式">
**何时更新支付方式：**
- 客户的卡过期或被替换
- 客户希望使用不同的支付方式
- 订阅因支付失败而处于 `on_hold` 状态
- 由于任何原因需要更改支付方式

```javascript Node.js
// Update payment method for active subscription
await client.subscriptions.updatePaymentMethod('sub_123', {
  type: 'new',
  return_url: 'https://example.com/return'
});

// Or use existing payment method
await client.subscriptions.updatePaymentMethod('sub_123', {
  type: 'existing',
  payment_method_id: 'pm_abc123'
});

// For on_hold subscription - automatically creates charge
const response = await client.subscriptions.updatePaymentMethod('sub_123', {
  type: 'new',
  return_url: 'https://example.com/return'
});

if (response.payment_id) {
  // Charge created for remaining dues
  // Redirect customer to response.payment_link
  // Monitor webhooks: payment.succeeded → subscription.active
}
```
