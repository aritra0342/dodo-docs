## 概述

按需订阅允许您一次授权客户的支付方式，然后在需要时收取可变金额，而不是按照固定的时间表进行收费。此功能适用于所有账户，无需批准。

使用本指南：
- 创建按需订阅（授权一个可选的初始价格的授权）
- 使用自定义金额触发后续收费
- 使用 Webhook 跟踪结果

有关一般订阅设置，请参见 [订阅集成指南](/developer-resources/subscription-integration-guide)。

## 先决条件

- Dodo Payments 商户账户和 API 密钥
- 配置的 Webhook 密钥和接收事件的端点
- 您目录中的订阅产品

<Tip>
如果你希望客户通过托管结账批准授权，请设置 `payment_link: true` 并提供 `return_url`。
</Tip>

## 按需工作原理

1. 你使用 `on_demand` 对象创建订阅，用于授权付款方式并可选地收取初始扣款。
2. 稍后，你通过专用扣款端点对该订阅创建自定义金额的扣款。
3. 你监听 webhooks（例如 `payment.succeeded`、`payment.failed`）以更新你的系统。

### 创建按需订阅

<Tabs>
<Tab title="Node.js SDK">

```javascript
import DodoPayments from 'dodopayments';

const client = new DodoPayments({
  bearerToken: process.env.DODO_PAYMENTS_API_KEY,
  environment: 'test_mode', // defaults to 'live_mode'
});

async function main() {
  const subscription = await client.checkoutSessions.create({
    product_cart: [{ product_id: 'pdt_123', quantity: 1 }],
    billing_address:  { city: 'SF', country: 'US', state: 'CA', street: '1 Market St', zipcode: '94105' },
    customer: { customer_id: 'cus_123' },
    return_url: 'https://example.com/billing/success',
    subscription_data: {
        on_demand: {
            mandate_only: true // set false to collect an initial charge
            // product_price: 1000, // optional: charge $10.00 now if mandate_only is false
            // product_currency: 'USD',
            // product_description: 'Custom initial charge',
            // adaptive_currency_fees_inclusive: false,
        }
    }
  });

  console.log(subscription.checkout_url);
}

main().catch(console.error);
```

import (

                // ProductDescription: "Custom initial charge",
                // AdaptiveCurrencyFeesInclusive: false,

端点：[POST /subscriptions/\{subscription_id\}/charge](/api-reference/subscriptions/create-charge)

请求的关键字段（主体）：

## 付款重试

我们的欺诈检测系统可能会阻止激进的重试模式（并可能将其标记为潜在的卡片测试）。遵循安全的重试策略。

<Warning>
突发重试模式可能会被我们的风控系统和处理方标记为欺诈或可疑的卡测试。避免集中重试；遵循下面的退避计划及时间对齐指导。
</Warning>

### 安全重试策略的原则

- **退避机制**：重试之间使用指数退避。
- **重试限制**：将总重试次数上限控制在 3 至 4 次。
- **智能过滤**：仅在可重试的失败（例如网络/发卡行错误、资金不足）上重试；切勿重试硬拒绝。
- **防止卡测试**：不要重试类似 `DO_NOT_HONOR`、`STOLEN_CARD`、`LOST_CARD`、`PICKUP_CARD`、`FRAUDULENT`、`AUTHENTICATION_FAILURE` 的失败。
- **变更元数据（可选）**：如果你维护自己的重试系统，可通过元数据（例如 `retry_attempt`）区分重试。

### 建议的重试计划（订阅）

- **第一次尝试**：创建收费时立即进行
- **第二次尝试**：3 天后
- **第三次尝试**：再过 7 天（总共 10 天）
- **第四次尝试（最终）**：再过 7 天（总共 17 天）

最后一步：如果仍未付款，请根据您的政策将订阅标记为未付款或取消它。在更新其支付方式的窗口期间通知客户。

### 避免突发重试；与授权时间对齐

- 将重试固定在原始授权时间戳上，以避免整个组合中出现“突发”行为。
- 示例：如果客户今天 13:10 开始试用或授权，则按退避计划在后续日期的 13:10 安排后续重试（例如 +3 天 → 13:10，+7 天 → 13:10）。
- 或者，如果你存储了最后一次成功付款时间 `T`，可将下一次尝试安排在 `T + X days`，以保持时间对齐。

<Note>
时区和夏令时：为调度使用一致的时间标准，仅为展示目的进行转换，以保持时间间隔。
</Note>

### 不应重试的拒绝代码

- `STOLEN_CARD`
- `DO_NOT_HONOR`
- `FRAUDULENT`
- `PICKUP_CARD`
- `AUTHENTICATION_FAILURE`
- `LOST_CARD`

<Note>
有关拒付原因的完整列表以及是否可由用户纠正，请参阅[交易失败](/api-reference/transaction-failures)文档。
</Note>

<Tip>
仅在软/临时问题（例如 `insufficient_funds`、`issuer_unavailable`、`processing_error`、网络超时）时重试。如果同样的拒付重复出现，请暂停进一步重试。
</Tip>

### 实施指南（无代码）

- 使用可持久化精确时间戳的调度器/队列；按相同的 HH:MM 计算下一次尝试的确切时间（例如 `T + 3 days`）。
- 保持并参考最后一次成功付款的时间戳 `T` 来计算下一次尝试；不要让多个订阅在同一瞬间集中。
- 始终评估最后一次拒付原因；若属于上述跳过列表中的硬拒绝，请停止重试。
- 限制每位客户和每个账户的并发重试，以防止意外激增。
- 主动沟通：在下次计划尝试前通过电子邮件/SMS 通知客户更新他们的付款方式。
- 仅将元数据用于可观察性（例如 `retry_attempt`）；切勿通过旋转无关字段来“规避”欺诈/风控系统。

## 使用 Webhook 跟踪结果

实现 Webhook 处理以跟踪客户旅程。请参见 [实现 Webhook](/developer-resources/integration-guide#implementing-webhooks)。

- **subscription.active**：授权的授权和激活的订阅
- **subscription.failed**：创建失败（例如，授权失败）
- **subscription.on_hold**：订阅被搁置（例如，未付款状态）
- **payment.succeeded**：收费成功
- **payment.failed**：收费失败

<Tip>
对于按需流程，请侧重 `payment.succeeded` 和 `payment.failed` 以对照使用量计费。
</Tip>

## 故障排除

- **422 无效请求**：确保在创建时提供 `on_demand.mandate_only`，并在扣款时提供 `product_price`。
- **货币错误**：如果你覆盖 `product_currency`，请确认它在你的账户和客户中受支持。
- **未收到 webhook**：核实你的 webhook URL 及签名密钥配置。
